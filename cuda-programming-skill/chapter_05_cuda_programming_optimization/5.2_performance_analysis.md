# 5.2 性能分析

## 一、本节作用

本节用于整理 CUDA 程序性能分析的方法和流程。

CUDA 程序优化不能只凭经验修改代码，而应先进行测时和性能分析，判断程序的主要瓶颈在哪里，再决定优化方向。否则可能出现代码变复杂但性能没有提升，甚至性能下降的情况。

CUDA 程序性能分析主要解决以下问题：

1. CUDA 程序到底比 CPU 串行程序快多少。
2. 总时间中有多少花在 host-device 数据传输上。
3. 有多少时间花在 kernel 执行上。
4. kernel 是 memory-bound 还是 compute-bound。
5. global memory 访问是否高效。
6. shared memory 是否存在 bank conflict。
7. block 和 thread 划分是否合理。
8. SM 利用率是否足够。
9. 是否存在 warp divergence。
10. 是否存在过多同步、原子操作或函数调用开销。

本节是第 5 章后续优化内容的入口。后续的存储器访问优化、任务划分优化和指令优化，都应该建立在性能分析结果之上。

---

## 二、CUDA 性能优化的基本目标

CUDA 程序优化的最终目标是提高程序加速比。

加速比通常定义为：

```text
speedup = cpu_time / cuda_time
```

其中：

- `cpu_time` 表示 CPU 串行程序执行时间。
- `cuda_time` 表示 CUDA 并行程序执行时间。

需要注意，`cuda_time` 可以有不同含义：

1. 只包含 kernel 执行时间。
2. 包含 host-to-device 数据传输、kernel 执行和 device-to-host 数据传输。
3. 包含完整程序时间，包括内存分配、初始化、数据传输、kernel 执行、结果验证等。

在实验报告中必须说明测量的时间范围，否则加速比没有可比性。

---

## 三、性能分析的三个维度

CUDA 程序性能可以从三个维度理解：

1. 加速比。
2. 资源利用率。
3. 指令效率。

---

## 3.1 加速比

加速比反映 CUDA 程序相对于 CPU 串行程序的整体提升。

公式：

```text
speedup = serial_time / parallel_time
```

例如：

```text
CPU 串行时间：100 ms
CUDA 总时间：10 ms
加速比：10×
```

如果只比较 kernel 时间，可能得到更高加速比；如果包含数据传输，实际加速比可能下降。因此需要明确说明：

```text
该加速比是否包含 host-device 数据传输。
```

---

## 3.2 资源利用率

资源利用率主要关注 GPU 硬件是否被充分使用。

常见关注点包括：

1. SM 利用率。
2. occupancy。
3. global memory 带宽利用率。
4. shared memory 使用效率。
5. register 使用情况。
6. active warps 数量。
7. kernel 并行度是否足够。

如果 GPU 的 SM 大量空闲，即使每个线程中的代码写得很简单，整体性能也可能不好。

---

## 3.3 指令效率

指令效率关注每条指令是否有效。

常见问题包括：

1. warp divergence。
2. 不必要的分支判断。
3. 不必要的 `__syncthreads()`。
4. 过多原子操作。
5. 使用低效数学函数。
6. 重复计算。
7. 寄存器溢出到 local memory。
8. global memory 非合并访问导致大量无效访存事务。

CUDA 优化不只是“增加线程数量”，还需要让每个线程执行更有效的指令流。

---

## 四、性能分析前需要明确的测时范围

在分析 CUDA 程序性能前，必须明确测量范围。

常见测时范围如下：

| 测时范围 | 包含内容 | 适用场景 |
|---|---|---|
| kernel time | 只包含 kernel 执行 | 分析 GPU 计算核心性能 |
| transfer time | H2D / D2H 数据传输 | 分析 PCIe 或数据搬移瓶颈 |
| CUDA total time | 数据传输 + kernel | 比较 CUDA 实际计算流程 |
| end-to-end time | 初始化 + 分配 + 传输 + kernel + 回传 + 验证 | 用户视角完整耗时 |
| CPU serial time | CPU 串行算法 | 计算加速比 |

建议在实验报告中至少给出：

```text
CPU 串行时间
H2D 数据传输时间
kernel 执行时间
D2H 数据传输时间
CUDA 总时间
加速比
```

这样才能判断程序到底是计算瓶颈，还是数据传输瓶颈。

## 五、CPU 计时器测时

## 5.1 CPU 计时器的作用

CPU 计时器可以测量 host 端看到的程序执行时间。

常见 C++ 计时方式：

```cpp
#include <chrono>
```

示例：

```cpp
auto start = std::chrono::high_resolution_clock::now();

// code to measure

auto stop = std::chrono::high_resolution_clock::now();

double milliseconds =
    std::chrono::duration<double, std::milli>(stop - start).count();
```

CPU 计时器适合测量：

1. 完整程序时间。
2. host 端数据初始化时间。
3. `cudaMemcpy` 同步传输时间。
4. 包含数据传输和 kernel 的 CUDA 总时间。
5. CPU 串行算法时间。

---

## 5.2 CPU 计时器测 CUDA kernel 的注意事项

CUDA kernel launch 通常是异步的。

也就是说，host 端执行：

```cpp
kernel<<<grid, block>>>(...);
```

之后，CPU 线程通常不会等待 kernel 完成，而是继续执行后续语句。

因此，下面这种测时是错误或不准确的：

```cpp
auto start = std::chrono::high_resolution_clock::now();

kernel<<<grid, block>>>(...);

auto stop = std::chrono::high_resolution_clock::now();
```

它测到的可能主要是 kernel 启动开销，而不是 kernel 实际执行时间。

正确方式是在结束计时前同步：

```cpp
auto start = std::chrono::high_resolution_clock::now();

kernel<<<grid, block>>>(...);
cudaDeviceSynchronize();

auto stop = std::chrono::high_resolution_clock::now();
```

教材中使用旧式 `cudaThreadSynchronize()`，现代 CUDA 程序中通常使用：

```cpp
cudaDeviceSynchronize()
```

---

## 5.3 CPU 计时器示例

```cpp
auto start = std::chrono::high_resolution_clock::now();

kernel<<<grid, block>>>(d_A, d_B, d_C, n);
CUDA_CHECK(cudaGetLastError());
CUDA_CHECK(cudaDeviceSynchronize());

auto stop = std::chrono::high_resolution_clock::now();

double kernel_ms =
    std::chrono::duration<double, std::milli>(stop - start).count();

printf("Kernel time measured by CPU timer: %.3f ms\n", kernel_ms);
```

这种方式可以使用，但对于 kernel 精确测时，更推荐使用 CUDA event。

---

## 六、CUDA event 测时

## 6.1 CUDA event 的作用

CUDA event 是 CUDA 中常用的 GPU 端计时工具。它可以记录 GPU stream 中某个时间点，并计算两个 event 之间的 GPU 执行时间。

CUDA event 适合测量：

1. kernel 执行时间。
2. 同一 stream 中多个 kernel 的总时间。
3. 异步数据传输时间。
4. 数据传输与 kernel 组合流程时间。

CUDA event 测得的是 GPU 时间线上的耗时，通常比 CPU 计时器更适合测量 kernel 执行时间。

---

## 6.2 CUDA event 基本用法

基本流程：

```text
1. 创建 start 和 stop event
2. 记录 start event
3. 执行待测 CUDA 操作
4. 记录 stop event
5. 同步 stop event
6. 计算 elapsed time
7. 销毁 event
```

示例：

```cpp
cudaEvent_t start, stop;

CUDA_CHECK(cudaEventCreate(&start));
CUDA_CHECK(cudaEventCreate(&stop));

CUDA_CHECK(cudaEventRecord(start));

kernel<<<grid, block>>>(d_A, d_B, d_C, n);
CUDA_CHECK(cudaGetLastError());

CUDA_CHECK(cudaEventRecord(stop));
CUDA_CHECK(cudaEventSynchronize(stop));

float milliseconds = 0.0f;
CUDA_CHECK(cudaEventElapsedTime(&milliseconds, start, stop));

CUDA_CHECK(cudaEventDestroy(start));
CUDA_CHECK(cudaEventDestroy(stop));

printf("Kernel time: %.3f ms\n", milliseconds);
```

---

## 6.3 在指定 stream 中使用 event

如果 kernel 在指定 stream 中执行，event 也应记录在同一个 stream 中。

```cpp
cudaStream_t stream;
CUDA_CHECK(cudaStreamCreate(&stream));

cudaEvent_t start, stop;
CUDA_CHECK(cudaEventCreate(&start));
CUDA_CHECK(cudaEventCreate(&stop));

CUDA_CHECK(cudaEventRecord(start, stream));

kernel<<<grid, block, 0, stream>>>(...);

CUDA_CHECK(cudaEventRecord(stop, stream));
CUDA_CHECK(cudaEventSynchronize(stop));

float ms = 0.0f;
CUDA_CHECK(cudaEventElapsedTime(&ms, start, stop));

CUDA_CHECK(cudaEventDestroy(start));
CUDA_CHECK(cudaEventDestroy(stop));
CUDA_CHECK(cudaStreamDestroy(stream));
```

这样测得的是该 stream 中两个 event 之间的 CUDA 操作耗时。

---

## 6.4 CUDA event 测时注意事项

使用 CUDA event 时需要注意：

1. kernel 启动后应使用 `cudaGetLastError()` 检查启动错误。
2. 需要使用 `cudaEventSynchronize(stop)` 等待 stop event 完成。
3. 测量结果通常以毫秒为单位。
4. 第一次 kernel 启动可能包含上下文初始化开销。
5. 正式测量前可以先 warm-up。
6. 对短 kernel 应重复执行多次再取平均值。
7. 如果测量异步拷贝，应配合 pinned memory 和 stream。
8. 不要把 CPU 端代码放在 CUDA event 之间期望被 event 计入。

---

## 七、warm-up 与多次测量

## 7.1 为什么需要 warm-up

CUDA 程序第一次运行 kernel 时，可能包含额外开销，例如：

1. CUDA context 初始化。
2. JIT 编译。
3. 缓存初始化。
4. 驱动层准备工作。

如果把第一次执行时间直接作为性能结果，可能偏大。

因此正式测量前可以先运行一次或多次 warm-up：

```cpp
kernel<<<grid, block>>>(...);
CUDA_CHECK(cudaGetLastError());
CUDA_CHECK(cudaDeviceSynchronize());
```

---

## 7.2 多次测量取平均值

对于执行时间较短的 kernel，单次测量波动较大。可以重复执行多次后取平均值。

示例：

```cpp
const int repeat = 100;

CUDA_CHECK(cudaEventRecord(start));

for (int i = 0; i < repeat; ++i) {
    kernel<<<grid, block>>>(...);
}

CUDA_CHECK(cudaEventRecord(stop));
CUDA_CHECK(cudaEventSynchronize(stop));

float total_ms = 0.0f;
CUDA_CHECK(cudaEventElapsedTime(&total_ms, start, stop));

float avg_ms = total_ms / repeat;
```

注意：如果 kernel 会修改输入数据，重复执行前需要保证数据状态合理。

---

## 八、加速比计算

## 8.1 基本加速比

```text
speedup = cpu_time / cuda_time
```

示例：

```cpp
double speedup = cpu_ms / cuda_ms;
```

输出：

```cpp
printf("Speedup: %.2fx\n", speedup);
```

---

## 8.2 kernel-only 加速比

如果只比较 CPU 计算时间和 GPU kernel 时间：

```text
kernel_speedup = cpu_compute_time / kernel_time
```

这种加速比通常较高，但没有包含 host-device 数据传输。

适合分析：

```text
GPU 计算核心相对于 CPU 计算核心的加速效果。
```

---

## 8.3 end-to-end 加速比

如果比较 CPU 完整程序和 CUDA 完整流程：

```text
end_to_end_speedup = cpu_total_time / cuda_total_time
```

其中 CUDA 总时间可以包含：

1. host-to-device 数据传输。
2. kernel 执行。
3. device-to-host 数据传输。

这种加速比更接近用户实际感受到的性能提升。

---

## 8.4 实验报告中的加速比写法

推荐写法：

```text
本文分别统计 CPU 串行算法时间、CUDA kernel 时间和 CUDA 总时间。
CUDA 总时间包含 host-to-device 数据传输、kernel 执行和 device-to-host 数据传输。
加速比使用 CPU 串行时间除以 CUDA 总时间得到。
```

不推荐写法：

```text
CUDA 加速比为 100 倍。
```

因为没有说明是否包含数据传输，也没有说明测时范围。

---

## 九、有效带宽计算

对于 memory-bound kernel，仅测量时间还不够，还应计算有效带宽。

有效带宽：

```text
effective_bandwidth = data_bytes / time_seconds
```

通常使用 GB/s 表示：

```text
GB/s = data_bytes / time_seconds / 1e9
```

---

## 9.1 向量加法带宽

向量加法：

```cpp
C[i] = A[i] + B[i];
```

每个元素通常涉及：

1. 读取 `A[i]`。
2. 读取 `B[i]`。
3. 写入 `C[i]`。

如果数据类型为 `float`，每个元素访问字节数大约为：

```text
3 × sizeof(float) = 12 bytes
```

总访问字节数：

```text
bytes = 3 × n × sizeof(float)
```

有效带宽：

```cpp
double gbps = (3.0 * n * sizeof(float)) / (kernel_ms / 1000.0) / 1e9;
```

---

## 9.2 为什么带宽重要

如果一个 kernel 主要做简单加法、复制、缩放、滤波等操作，它可能不是计算瓶颈，而是访存瓶颈。

例如：

```cpp
C[idx] = A[idx] + B[idx];
```

每个元素只做一次加法，却需要多次 global memory 访问。此类 kernel 的性能通常更受内存带宽限制。

分析这类程序时，应重点关注：

1. global memory 是否合并访问。
2. 是否存在重复读取。
3. 是否可以使用 shared memory 缓存。
4. 是否可以减少 host-device 数据传输。
5. 实测带宽是否远低于硬件理论带宽。

---

## 十、计算吞吐量估算

对于 compute-bound kernel，可以估算浮点运算吞吐量。

公式：

```text
GFLOPS = floating_point_operations / time_seconds / 1e9
```

例如，一个 kernel 总共执行 `2 * n` 次浮点操作，执行时间为 `kernel_ms`：

```cpp
double gflops = (2.0 * n) / (kernel_ms / 1000.0) / 1e9;
```

需要注意，浮点操作次数必须根据算法实际估算。不要在没有依据的情况下随意写 GFLOPS。

## 十一、CUDA 程序性能分析工具

## 11.1 教材中的工具

教材中介绍的工具包括：

1. CUDA Profiler。
2. Nvidia Parallel Nsight。

这些工具对应较早版本的 CUDA 开发环境，对于理解 CUDA 性能分析思想仍有价值。

它们关注的问题包括：

1. kernel 执行时间。
2. memory copy 时间。
3. kernel launch 时间。
4. 存储器访问效率。
5. SM 利用率。
6. 指令执行情况。
7. 程序性能瓶颈。

---

## 11.2 现代 CUDA 中的工具

现代 CUDA 开发中，更常用：

1. Nsight Systems。
2. Nsight Compute。

其中：

```text
Nsight Systems：看整体时间线
Nsight Compute：看单个 kernel 的详细指标
```

---

## 11.3 Nsight Systems

Nsight Systems 适合进行系统级性能分析。

它主要回答：

```text
程序时间都花在哪里？
CPU 和 GPU 是否重叠执行？
数据传输是否占据大量时间？
kernel 之间是否存在空隙？
stream 是否真正并发？
```

常见用途：

1. 查看 CPU / GPU 时间线。
2. 分析 kernel launch 开销。
3. 分析 host-device 数据传输。
4. 分析 stream 并发。
5. 分析 CPU 是否频繁等待 GPU。
6. 判断是否存在大量空闲时间。

命令示例：

```bash
nsys profile -t cuda,nvtx,osrt -o report ./main
```

生成报告后可以使用图形界面打开：

```bash
nsys-ui report.nsys-rep
```

也可以导出统计信息：

```bash
nsys stats report.nsys-rep
```

---

## 11.4 Nsight Compute

Nsight Compute 适合分析单个 CUDA kernel 的详细性能。

它主要回答：

```text
这个 kernel 为什么慢？
是访存瓶颈、计算瓶颈、occupancy 不足，还是分支发散？
```

常见关注指标包括：

1. SM 利用率。
2. achieved occupancy。
3. memory throughput。
4. global memory load/store efficiency。
5. shared memory bank conflict。
6. register 使用数量。
7. local memory 使用情况。
8. warp execution efficiency。
9. branch efficiency。
10. 指令吞吐量。
11. source line 对应的性能热点。

命令示例：

```bash
ncu ./main
```

收集较完整指标：

```bash
ncu --set full ./main
```

保存报告：

```bash
ncu --set full -o report ./main
```

只分析特定 kernel：

```bash
ncu --kernel-name vectorAddKernel ./main
```

---

## 11.5 编译时添加 `-lineinfo`

为了让性能分析工具把热点对应到源代码行，建议编译时加入：

```bash
nvcc -O2 -lineinfo main.cu -o main
```

`-lineinfo` 通常比 `-G` 更适合性能分析。

不要用下面命令做正式性能测试：

```bash
nvcc -g -G main.cu -o main_debug
```

因为 `-G` 会影响 device 端优化，导致性能结果不能代表 release 版本。

## 十二、性能分析的基本流程

CUDA 程序性能分析可以按照以下流程进行。

---

## 12.1 建立正确性基线

优化前必须先保证程序结果正确。

基本步骤：

1. 编写 CPU 串行版本。
2. 编写 CUDA 基础版本。
3. 对比 CPU 和 CUDA 输出。
4. 设置误差容忍范围。
5. 确保不同数据规模下结果正确。

如果程序结果不正确，性能再高也没有意义。

---

## 12.2 建立性能基线

先测量未优化版本性能。

需要记录：

1. 输入规模。
2. GPU 型号。
3. CUDA 版本。
4. 编译命令。
5. block 和 grid 配置。
6. CPU 串行时间。
7. H2D 时间。
8. kernel 时间。
9. D2H 时间。
10. CUDA 总时间。
11. 加速比。

建议形成表格：

| 指标 | 数值 |
|---|---|
| 数据规模 | `n = ...` |
| block size | `256` |
| CPU 时间 | `... ms` |
| H2D 时间 | `... ms` |
| kernel 时间 | `... ms` |
| D2H 时间 | `... ms` |
| CUDA 总时间 | `... ms` |
| 加速比 | `...×` |

---

## 12.3 拆分时间组成

将 CUDA 总时间拆分为：

```text
CUDA total time = H2D time + kernel time + D2H time
```

如果 H2D 和 D2H 占比很高，说明主要瓶颈可能是数据传输。

如果 kernel 占比很高，说明主要瓶颈在 GPU 计算或访存。

示例判断：

```text
H2D + D2H 占总时间 70%：优先减少数据传输
kernel 占总时间 90%：优先优化 kernel
```

---

## 12.4 使用 Nsight Systems 看整体时间线

使用 Nsight Systems 检查：

1. CPU 是否在等待 GPU。
2. H2D、kernel、D2H 是否串行。
3. 是否存在大量小 kernel。
4. 是否频繁调用 `cudaMemcpy`。
5. 是否存在 GPU 空闲间隙。
6. stream 是否有重叠效果。
7. 是否可以使用 pinned memory 和异步复制。

---

## 12.5 使用 Nsight Compute 看 kernel 细节

使用 Nsight Compute 检查：

1. global memory 是否合并访问。
2. memory throughput 是否接近硬件上限。
3. shared memory 是否有 bank conflict。
4. register 是否过多导致 occupancy 降低。
5. 是否存在 local memory 访问。
6. warp divergence 是否严重。
7. branch efficiency 是否较低。
8. 指令吞吐量是否较低。
9. 是否存在过多原子操作。
10. 是否存在同步开销。

---

## 12.6 一次只优化一个因素

性能优化时，不建议一次修改大量内容。

推荐流程：

```text
测量 baseline
    ↓
发现一个瓶颈
    ↓
只修改一个优化点
    ↓
重新测量
    ↓
对比结果
    ↓
保留有效优化，撤销无效优化
```

这样才能判断每个优化措施是否真的有效。

## 十三、常见性能瓶颈判断

| 现象 | 可能瓶颈 | 优先检查 |
|---|---|---|
| H2D / D2H 时间占比高 | 数据传输瓶颈 | 减少传输、pinned memory、stream |
| kernel 时间长，memory throughput 高 | memory-bound | 合并访问、shared memory、减少访存 |
| kernel 时间长，计算吞吐低 | compute-bound | 指令优化、数学函数、减少重复计算 |
| SM 利用率低 | 并行度不足 | block/grid 配置、任务划分 |
| achieved occupancy 低 | 资源限制 | register、shared memory、block size |
| warp execution efficiency 低 | 分支发散 | 控制流优化 |
| shared memory bank conflict 高 | shared memory 访问冲突 | padding、访问模式 |
| local memory 使用多 | register spill 或局部数组过大 | 减少局部变量、优化数组 |
| kernel 很短但很多 | launch overhead 高 | kernel fusion、批处理 |
| 原子操作占比高 | atomic bottleneck | 分层归约、减少冲突 |

---

## 十四、优化方向与分析依据

## 14.1 数据传输瓶颈

如果数据传输占比较高，可以考虑：

1. 减少 host-device 数据传输次数。
2. 将中间结果保留在 device memory。
3. 合并小数据传输为大块传输。
4. 使用 pinned memory。
5. 使用 `cudaMemcpyAsync`。
6. 使用 stream 重叠传输和计算。
7. 判断是否适合 zero-copy。

典型低效流程：

```text
H2D -> kernel1 -> D2H
H2D -> kernel2 -> D2H
H2D -> kernel3 -> D2H
```

更好流程：

```text
H2D -> kernel1 -> kernel2 -> kernel3 -> D2H
```

---

## 14.2 global memory 瓶颈

如果 kernel 主要受 global memory 限制，可以考虑：

1. 保证连续线程访问连续地址。
2. 使用 coalesced access。
3. 减少重复读取。
4. 使用 shared memory 缓存可复用数据。
5. 使用 constant memory 保存小规模只读参数。
6. 使用 texture memory 处理空间局部只读访问。
7. 调整 AoS 为 SoA。
8. 使用 pitch memory 处理二维对齐。

---

## 14.3 shared memory 瓶颈

如果 shared memory 使用后性能不升反降，应检查：

1. 数据是否真的存在复用。
2. 是否引入过多 `__syncthreads()`。
3. 是否存在 bank conflict。
4. shared memory 使用量是否过大。
5. occupancy 是否下降。
6. global-to-shared 拷贝是否额外增加开销。

---

## 14.4 任务划分瓶颈

如果 SM 利用率低或 occupancy 不足，应检查：

1. block size 是否合理。
2. block 数量是否足够。
3. 每个 block 是否使用过多 shared memory。
4. 每个线程是否使用过多 register。
5. 数据规模是否太小。
6. 是否存在负载不均衡。
7. 是否有线程大量空闲。

常见 block size：

```text
128
256
512
```

通常应选择 32 的倍数，因为 warp 大小通常为 32。

---

## 14.5 指令瓶颈

如果 kernel 不是访存瓶颈，而是指令执行效率低，可以考虑：

1. 减少分支发散。
2. 减少循环中的重复计算。
3. 使用更合适的数据类型。
4. 避免昂贵数学函数。
5. 谨慎使用 `--use_fast_math`。
6. 减少原子操作。
7. 减少不必要的同步。
8. 使用更简单的索引计算。

---

## 十五、完整示例：CPU 计时 + CUDA event 计时

下面示例展示如何同时测量：

1. CPU 串行时间。
2. H2D 数据传输时间。
3. kernel 执行时间。
4. D2H 数据传输时间。
5. CUDA 总时间。
6. 加速比。
7. kernel 有效带宽。

```cpp
#include <cstdio>
#include <cstdlib>
#include <cmath>
#include <chrono>
#include <cuda_runtime.h>

#define CUDA_CHECK(call)                                                   \
    do {                                                                   \
        cudaError_t err = (call);                                           \
        if (err != cudaSuccess) {                                           \
            fprintf(stderr, "CUDA error at %s:%d: %s\n",                   \
                    __FILE__, __LINE__, cudaGetErrorString(err));           \
            exit(EXIT_FAILURE);                                             \
        }                                                                  \
    } while (0)

__global__ void vectorAddKernel(const float* A,
                                const float* B,
                                float* C,
                                int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    if (idx < n) {
        C[idx] = A[idx] + B[idx];
    }
}

void vectorAddCPU(const float* A, const float* B, float* C, int n) {
    for (int i = 0; i < n; ++i) {
        C[i] = A[i] + B[i];
    }
}

int main() {
    const int n = 1 << 24;
    const size_t bytes = n * sizeof(float);

    float* h_A = static_cast<float*>(malloc(bytes));
    float* h_B = static_cast<float*>(malloc(bytes));
    float* h_C = static_cast<float*>(malloc(bytes));
    float* h_C_ref = static_cast<float*>(malloc(bytes));

    if (h_A == nullptr || h_B == nullptr ||
        h_C == nullptr || h_C_ref == nullptr) {
        fprintf(stderr, "Host allocation failed.\n");
        return EXIT_FAILURE;
    }

    for (int i = 0; i < n; ++i) {
        h_A[i] = static_cast<float>(i);
        h_B[i] = static_cast<float>(2 * i);
        h_C[i] = 0.0f;
        h_C_ref[i] = 0.0f;
    }

    auto cpu_start = std::chrono::high_resolution_clock::now();

    vectorAddCPU(h_A, h_B, h_C_ref, n);

    auto cpu_stop = std::chrono::high_resolution_clock::now();

    double cpu_ms =
        std::chrono::duration<double, std::milli>(cpu_stop - cpu_start).count();

    float* d_A = nullptr;
    float* d_B = nullptr;
    float* d_C = nullptr;

    CUDA_CHECK(cudaMalloc((void**)&d_A, bytes));
    CUDA_CHECK(cudaMalloc((void**)&d_B, bytes));
    CUDA_CHECK(cudaMalloc((void**)&d_C, bytes));

    dim3 block(256);
    dim3 grid((n + block.x - 1) / block.x);

    // warm-up
    CUDA_CHECK(cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice));
    CUDA_CHECK(cudaMemcpy(d_B, h_B, bytes, cudaMemcpyHostToDevice));
    vectorAddKernel<<<grid, block>>>(d_A, d_B, d_C, n);
    CUDA_CHECK(cudaGetLastError());
    CUDA_CHECK(cudaDeviceSynchronize());

    auto cuda_total_start = std::chrono::high_resolution_clock::now();

    auto h2d_start = std::chrono::high_resolution_clock::now();

    CUDA_CHECK(cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice));
    CUDA_CHECK(cudaMemcpy(d_B, h_B, bytes, cudaMemcpyHostToDevice));

    auto h2d_stop = std::chrono::high_resolution_clock::now();

    double h2d_ms =
        std::chrono::duration<double, std::milli>(h2d_stop - h2d_start).count();

    cudaEvent_t start, stop;
    CUDA_CHECK(cudaEventCreate(&start));
    CUDA_CHECK(cudaEventCreate(&stop));

    CUDA_CHECK(cudaEventRecord(start));

    vectorAddKernel<<<grid, block>>>(d_A, d_B, d_C, n);
    CUDA_CHECK(cudaGetLastError());

    CUDA_CHECK(cudaEventRecord(stop));
    CUDA_CHECK(cudaEventSynchronize(stop));

    float kernel_ms = 0.0f;
    CUDA_CHECK(cudaEventElapsedTime(&kernel_ms, start, stop));

    auto d2h_start = std::chrono::high_resolution_clock::now();

    CUDA_CHECK(cudaMemcpy(h_C, d_C, bytes, cudaMemcpyDeviceToHost));

    auto d2h_stop = std::chrono::high_resolution_clock::now();

    double d2h_ms =
        std::chrono::duration<double, std::milli>(d2h_stop - d2h_start).count();

    auto cuda_total_stop = std::chrono::high_resolution_clock::now();

    double cuda_total_ms =
        std::chrono::duration<double, std::milli>(
            cuda_total_stop - cuda_total_start
        ).count();

    bool correct = true;
    for (int i = 0; i < n; ++i) {
        if (fabs(h_C[i] - h_C_ref[i]) > 1e-5f) {
            correct = false;
            fprintf(stderr, "Mismatch at %d: got %f, expected %f\n",
                    i, h_C[i], h_C_ref[i]);
            break;
        }
    }

    double speedup_total = cpu_ms / cuda_total_ms;
    double speedup_kernel = cpu_ms / static_cast<double>(kernel_ms);

    double kernel_bandwidth_gbps =
        (3.0 * static_cast<double>(bytes)) /
        (static_cast<double>(kernel_ms) / 1000.0) / 1e9;

    printf("Result: %s\n", correct ? "PASS" : "FAIL");
    printf("CPU serial time: %.3f ms\n", cpu_ms);
    printf("H2D time: %.3f ms\n", h2d_ms);
    printf("Kernel time: %.3f ms\n", kernel_ms);
    printf("D2H time: %.3f ms\n", d2h_ms);
    printf("CUDA total time: %.3f ms\n", cuda_total_ms);
    printf("Kernel-only speedup: %.2fx\n", speedup_kernel);
    printf("End-to-end speedup: %.2fx\n", speedup_total);
    printf("Effective kernel bandwidth: %.2f GB/s\n", kernel_bandwidth_gbps);

    CUDA_CHECK(cudaEventDestroy(start));
    CUDA_CHECK(cudaEventDestroy(stop));

    CUDA_CHECK(cudaFree(d_A));
    CUDA_CHECK(cudaFree(d_B));
    CUDA_CHECK(cudaFree(d_C));

    free(h_A);
    free(h_B);
    free(h_C);
    free(h_C_ref);

    return correct ? EXIT_SUCCESS : EXIT_FAILURE;
}
```

编译命令：

```bash
nvcc -O2 -lineinfo vector_add_timing.cu -o vector_add_timing
```

运行命令：

```bash
./vector_add_timing
```

使用 Nsight Systems：

```bash
nsys profile -t cuda,nvtx,osrt -o vector_add_nsys ./vector_add_timing
```

使用 Nsight Compute：

```bash
ncu --set full -o vector_add_ncu ./vector_add_timing
```

## 十七、常见错误

## 17.1 用 CPU 计时器测 kernel 但没有同步

错误：

```cpp
auto start = std::chrono::high_resolution_clock::now();

kernel<<<grid, block>>>(...);

auto stop = std::chrono::high_resolution_clock::now();
```

正确：

```cpp
auto start = std::chrono::high_resolution_clock::now();

kernel<<<grid, block>>>(...);
CUDA_CHECK(cudaDeviceSynchronize());

auto stop = std::chrono::high_resolution_clock::now();
```

---

## 17.2 只报告 kernel 加速比

错误写法：

```text
CUDA 程序加速比为 100 倍。
```

但实际上只比较了 CPU 时间和 kernel 时间，没有包含数据传输。

更严谨写法：

```text
如果只统计 kernel 时间，加速比为 100 倍；如果统计 H2D、kernel 和 D2H 的 CUDA 总时间，加速比为 25 倍。
```

---

## 17.3 把数据初始化时间计入 CPU，但不计入 CUDA

比较 CPU 与 CUDA 时，应保证计时范围公平。

不公平比较：

```text
CPU 时间包含数据初始化
CUDA 时间不包含数据初始化
```

或：

```text
CPU 时间只包含计算
CUDA 时间包含传输和计算
```

应明确说明各自计时范围。

---

## 17.4 使用 debug 编译结果做性能结论

错误：

```bash
nvcc -g -G main.cu -o main
```

然后用该版本测性能。

`-G` 会影响 device 端优化，不适合正式性能测试。

性能测试建议：

```bash
nvcc -O2 -lineinfo main.cu -o main
```

---

## 17.5 没有 warm-up

如果第一次 kernel 执行包含 CUDA context 初始化开销，测得结果可能偏大。

建议：

```cpp
kernel<<<grid, block>>>(...);
cudaDeviceSynchronize();

// 正式计时
```

---

## 17.6 单次测量就下结论

性能测量可能有波动。对于短 kernel，应多次测量取平均值。

---

## 17.7 忽略数据规模

CUDA 程序在小数据规模下可能不快，因为 kernel 启动和数据传输开销相对明显。

应测试多个数据规模，例如：

```text
1K
16K
1M
16M
```

并分析加速比变化趋势。

---

## 17.8 不检查 kernel 错误

错误：

```cpp
kernel<<<grid, block>>>(...);
cudaDeviceSynchronize();
```

建议：

```cpp
kernel<<<grid, block>>>(...);
CUDA_CHECK(cudaGetLastError());
CUDA_CHECK(cudaDeviceSynchronize());
```

否则 kernel 启动错误可能被忽略，测得时间和结果都不可信。

---

## 17.9 优化前没有 baseline

如果没有保存优化前数据，就无法证明优化有效。

应记录：

```text
优化前时间
优化后时间
提升比例
修改内容
测试条件
```

---

## 17.10 只看时间不看瓶颈

时间下降或上升只是结果，还需要分析原因。

例如：

```text
使用 shared memory 后变慢
```

可能原因包括：

1. 数据没有复用。
2. 增加了同步开销。
3. 发生 bank conflict。
4. shared memory 使用过多导致 occupancy 降低。
5. global memory 原本已经合并访问。

## 十八、回答模板

## 18.1 解释 CUDA 程序如何计时

```text
CUDA 程序计时通常分为 CPU 计时器和 CUDA event 两类。CPU 计时器适合测量完整程序时间、数据传输时间和 host 端流程时间，但由于 kernel 启动是异步的，如果用 CPU 计时器测 kernel，需要在结束计时前调用 cudaDeviceSynchronize。CUDA event 更适合测量 GPU 时间线上的 kernel 执行时间，常用 cudaEventRecord、cudaEventSynchronize 和 cudaEventElapsedTime。
```

---

## 18.2 解释为什么 kernel 计时需要同步

```text
CUDA kernel 启动通常是异步的。CPU 发起 kernel launch 后，通常不会等待 GPU 执行完 kernel，而是继续执行后续 host 代码。因此，如果用 CPU 计时器测量 kernel 而不进行同步，测到的可能只是 kernel 启动开销。正确做法是在计时结束前调用 cudaDeviceSynchronize，或者使用 CUDA event 测量 GPU 端实际执行时间。
```

---

## 18.3 解释 CUDA event

```text
CUDA event 是 CUDA 中用于 GPU 时间线测量的事件对象。通常创建 start 和 stop 两个 event，在 kernel 前后分别调用 cudaEventRecord，然后使用 cudaEventSynchronize 等待 stop event 完成，最后用 cudaEventElapsedTime 得到两个 event 之间的耗时。CUDA event 通常用于精确测量 kernel 执行时间。
```

---

## 18.4 解释 Nsight Systems 和 Nsight Compute 的区别

```text
Nsight Systems 主要用于系统级时间线分析，适合查看 CPU、GPU、数据传输、kernel 启动和 stream 并发之间的关系。Nsight Compute 主要用于单个 CUDA kernel 的详细性能分析，适合查看 memory throughput、occupancy、warp divergence、shared memory bank conflict、register 使用等指标。简单来说，Nsight Systems 用来看整体时间花在哪里，Nsight Compute 用来看某个 kernel 为什么慢。
```

---

## 18.5 解释如何判断 CUDA 程序瓶颈

```text
判断 CUDA 程序瓶颈时，应先拆分 H2D 数据传输时间、kernel 时间和 D2H 数据传输时间。如果数据传输占比高，应优先减少 host-device 数据交换，或使用 pinned memory、cudaMemcpyAsync 和 stream。如果 kernel 时间占比高，应进一步分析 global memory 访问效率、SM 利用率、occupancy、分支发散、shared memory bank conflict 和指令吞吐量。优化应基于测量结果，而不是凭感觉修改代码。
```

---

## 18.6 解释 kernel-only 加速比和 end-to-end 加速比

```text
kernel-only 加速比只比较 CPU 串行计算时间和 GPU kernel 执行时间，反映 GPU 计算核心的加速效果。end-to-end 加速比则把 host-to-device 数据传输、kernel 执行和 device-to-host 数据传输都计入 CUDA 总时间，更接近用户实际感受到的性能提升。实验报告中应明确说明使用的是哪一种加速比。
```

## 二十、小结

CUDA 程序优化必须建立在性能分析基础之上。优化前应先明确测时范围，分别测量 CPU 串行时间、host-to-device 数据传输时间、kernel 执行时间、device-to-host 数据传输时间和 CUDA 总时间。只有这样，才能判断程序瓶颈到底在数据传输、global memory 访问、任务划分、指令效率还是同步开销。

CPU 计时器适合测量完整程序流程，但由于 CUDA kernel 启动通常是异步的，用 CPU 计时器测 kernel 时必须同步。CUDA event 更适合测量 GPU 端 kernel 执行时间，是 CUDA kernel 性能测试中最常用的方法之一。

现代 CUDA 性能分析通常使用 Nsight Systems 和 Nsight Compute。Nsight Systems 用于观察整体时间线，分析 CPU/GPU 并发、数据传输和 kernel 调度；Nsight Compute 用于分析单个 kernel 的详细性能指标，例如 memory throughput、occupancy、warp divergence、bank conflict 和 register 使用情况。

本节的核心结论是：不要凭感觉优化 CUDA 程序。正确流程应是先测时，拆分时间组成，定位瓶颈，再针对瓶颈选择优化方法，最后重新测量并比较优化前后的结果。
