# GPU并行与分布式集群并行选型准则

## 何时使用
- 面临大规模计算任务
- 需要在GPU并行与分布式集群并行之间做技术决策
- 已明确任务的数据规模、任务粒度及通信模式

> **注意**：本技能不适用于GPU集群等混合架构中的跨节点+节点内协同场景；若忽略具体硬件型号差异，可能导致误判。

## 执行步骤

### 1. 判断任务粒度与通信模式
- **粗粒度并行**：任务可分解为少量大型子任务，子任务间通信稀疏但单次数据量大。
- **细粒度并行**：任务包含大量小型计算单元（如像素、粒子、矩阵元素），每个单元计算简单但总数巨大。

### 2. 根据任务特征选择并行架构

#### 若为粗粒度、网络密集型任务（如大规模区域分解仿真）：
- 采用**分布式集群**（MIMD架构）
- 使用Master-Slave模式：Master分发任务块，Slave节点独立计算
- 节点间通过Infiniband或以太网通信，需优化负载均衡与通信开销

#### 若为细粒度、计算密集型任务（如图像处理、稠密矩阵运算）：
- 采用**GPU并行**（基于SIMT模型）
- 利用GPU轻量级线程创建（如warp=32线程）和零开销线程调度
- 尽量减少CPU-GPU间通过PCI-E的数据传输

### 3. 关键性能指标对比参考
- **通信带宽**：GPU内部存储带宽（如Tesla达177GB/s）远高于集群节点间（通常仅数GB/s）
- **能效比**：GPU提供更高每瓦特FLOPS，适合能效敏感场景
- **编程模型**：集群需显式消息传递（如MPI），GPU使用共享内存与同步原语

### 4. 输出结果
推荐采用GPU并行或分布式集群并行，并说明理由，包括任务特征匹配依据与预期性能优势。

> **例外提示**：若任务同时具备粗粒度（跨节点）和细粒度（节点内）特征，应考虑GPU集群混合架构——此情况超出本技能适用范围。
