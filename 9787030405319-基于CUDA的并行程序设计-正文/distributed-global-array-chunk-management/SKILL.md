# 分布式全局地址系统的数据分块管理机制

## 适用场景
- 处理大规模全局数组，其总大小超出单节点内存容量
- 执行核外计算任务（out-of-core computation）
- 需要GPU显存与主存协同工作的高性能计算场景

## 前提条件
- 已部署 DataCutter 中间件
- 系统支持段页式存储模型
- 节点配备外存设备（如 SSD）

## 核心流程

### 1. 定义全局数组结构体（段级管理）
创建包含以下字段的全局数组元数据：
- `global_size`：数组总长度（元素数量）
- `block_size`：每块的标准长度（可自定义）
- `number_of_block`：块总数，计算为 `ceil(global_size / block_size)`
- `divisibility`：分块完整性标志（0 表示整除，1 表示有余数）

### 2. 定义块数组结构体（页级管理）
每个数据块包含以下字段：
- `size`：实际长度（最后一块可能小于 `block_size`）
- `id`：块在全局数组中的序号
- `address`：主存物理地址指针（初始为空）
- `GPU_address`：GPU 显存地址
- `list_have`：本块中可用数据区间的链表
- `number_user`：当前使用该块的线程数（>0 时不可释放）
- `on_disk`：本节点外存是否存有该块（1/0）
- `somebody_have`：其他节点是否持有该块（1/0）
- `nodex`：持有该块的节点列表

### 3. 创建与映射
- 创建全局数组时，同时实例化全局数组结构和所有块结构
- 全局数组按名称注册到全局数组表
- 每个块按 `(数组名, id)` 注册到块数组表
- **延迟分配内存**：仅在首次请求某块数据时才分配主存或 GPU 显存

### 4. 处理数据请求
当应用程序请求区间 `{m, n}` 时：
- 系统自动计算该区间所属的块 ID
- 若块未加载，则从外存或远程节点加载
- 返回对应内存或 GPU 显存的物理地址
- 对用户透明处理外存交换和分布式数据定位

## 使用限制
- 不适用于小规模、可完全驻留内存的数据
- 必须配合分布式数据服务协议使用

## 触发条件
IF 需管理超出内存容量的全局数组并支持 GPU 加速 THEN 应用此分块管理机制
