# 4.5 主机端页锁定内存的使用技巧

## 一、本节作用

本节用于整理 CUDA 主机端页锁定内存 pinned memory 的使用方法和优化技巧。

在普通 CUDA 程序中，host 端数据通常使用 `malloc`、`new` 或 `std::vector` 分配。这类内存属于 pageable host memory，即可分页主机内存。可分页内存由操作系统管理，可能被换出到磁盘，因此 GPU 与这类内存进行数据传输时，通常需要经过额外的中间步骤。

pinned memory 也称 page-locked memory，即页锁定内存。它位于主机端物理内存中，不会被操作系统换出。CUDA 可以更高效地在 pinned memory 与 device memory 之间进行数据传输，并且 pinned memory 是实现 `cudaMemcpyAsync` 异步传输的重要前提。

本节重点解决以下问题：

1. 什么是 pinned memory。
2. pinned memory 与 pageable memory 有什么区别。
3. 如何使用 `cudaHostAlloc` 和 `cudaMallocHost` 分配 pinned memory。
4. 什么是 zero-copy。
5. 如何使用 mapped pinned memory。
6. 什么是 CUDA stream。
7. 如何使用 `cudaMemcpyAsync`。
8. 如何实现数据传输与 kernel 执行重叠。
9. pinned memory 的适用场景和风险。
10. 如何避免主机端页锁定内存使用中的常见错误。

---

## 二、pageable host memory 与 pinned host memory

CUDA 中 host 端内存主要可以分为两类：

1. pageable host memory
2. pinned host memory

---

## 2.1 pageable host memory

pageable host memory 是普通主机端内存，通常由操作系统管理。

常见分配方式：

```cpp
float* h_A = static_cast<float*>(malloc(bytes));
```

或：

```cpp
std::vector<float> h_A(n);
```

pageable host memory 的特点：

1. 分配方便。
2. 释放方便。
3. 由操作系统管理。
4. 可能被操作系统换出到磁盘。
5. 适合普通 CPU 程序。
6. 不适合高性能异步传输。
7. 与 GPU 进行数据传输时可能需要额外拷贝。

对于入门 CUDA 程序，使用 pageable memory 是可以的。例如向量加法、矩阵加法等基础实验都可以使用普通 `malloc` 或 `std::vector`。

---

## 2.2 pinned host memory

pinned host memory 是页锁定主机内存。它只分配在主机端物理内存中，并且不会参与操作系统的页面置换。

常见分配方式：

```cpp
cudaHostAlloc
cudaMallocHost
```

释放方式：

```cpp
cudaFreeHost
```

pinned host memory 的特点：

1. 位于 host 端物理内存。
2. 不会被操作系统换出。
3. 与 device memory 之间传输通常更高效。
4. 可以用于 `cudaMemcpyAsync` 异步传输。
5. 可以配合 stream 实现传输与计算重叠。
6. 可以用于 mapped memory 和 zero-copy。
7. 分配和释放成本高于普通内存。
8. 过量使用会降低系统整体性能。

---

## 2.3 pageable memory 与 pinned memory 对比

| 对比项 | pageable host memory | pinned host memory |
|---|---|---|
| 分配方式 | `malloc`、`new`、`std::vector` | `cudaHostAlloc`、`cudaMallocHost` |
| 释放方式 | `free`、`delete` | `cudaFreeHost` |
| 是否可被换出 | 可以 | 不可以 |
| 传输效率 | 通常较低 | 通常较高 |
| 是否支持真正异步传输 | 通常不适合 | 适合 |
| 分配成本 | 较低 | 较高 |
| 过量使用风险 | 较小 | 可能影响系统性能 |
| 典型用途 | 普通 host 数据 | 高速传输、异步传输、zero-copy |

简单判断：

```text
小规模数据、普通实验：pageable memory 即可
大规模传输、异步传输、stream 流水线：使用 pinned memory
```

## 三、pinned memory 的基本使用

## 3.1 使用 `cudaMallocHost`

`cudaMallocHost` 可以分配 pinned host memory。

示例：

```cpp
float* h_A = nullptr;
size_t bytes = n * sizeof(float);

cudaMallocHost((void**)&h_A, bytes);
```

释放：

```cpp
cudaFreeHost(h_A);
```

完整示例：

```cpp
float* h_A = nullptr;
const int n = 1024;
const size_t bytes = n * sizeof(float);

CUDA_CHECK(cudaMallocHost((void**)&h_A, bytes));

for (int i = 0; i < n; ++i) {
    h_A[i] = static_cast<float>(i);
}

CUDA_CHECK(cudaFreeHost(h_A));
```

`cudaMallocHost` 的作用类似于分配普通 host 数组，但返回的是页锁定主机内存。

---

## 3.2 使用 `cudaHostAlloc`

`cudaHostAlloc` 是更灵活的 pinned memory 分配函数，可以指定不同标志。

函数形式：

```cpp
cudaError_t cudaHostAlloc(void** pHost, size_t size, unsigned int flags);
```

基本用法：

```cpp
float* h_A = nullptr;
CUDA_CHECK(cudaHostAlloc((void**)&h_A, bytes, cudaHostAllocDefault));
```

释放：

```cpp
CUDA_CHECK(cudaFreeHost(h_A));
```

---

## 3.3 `cudaMallocHost` 与 `cudaHostAlloc`

`cudaMallocHost` 可以理解为较简单的 pinned memory 分配接口。

`cudaHostAlloc` 更灵活，可以通过 flags 控制内存属性。

| 函数 | 特点 |
|---|---|
| `cudaMallocHost` | 简单分配 pinned memory |
| `cudaHostAlloc` | 可指定 portable、mapped、write-combined 等属性 |

在普通异步传输场景中，二者都可以使用。

如果需要 zero-copy 或特殊属性，通常使用 `cudaHostAlloc`。

---

## 四、`cudaHostAlloc` 常用标志

`cudaHostAlloc` 的第三个参数可以指定 pinned memory 的属性。

常见标志包括：

```cpp
cudaHostAllocDefault
cudaHostAllocPortable
cudaHostAllocMapped
cudaHostAllocWriteCombined
```

---

## 4.1 `cudaHostAllocDefault`

默认页锁定内存。

```cpp
CUDA_CHECK(cudaHostAlloc((void**)&h_A, bytes, cudaHostAllocDefault));
```

适合普通高速传输和异步传输。

---

## 4.2 `cudaHostAllocPortable`

使用该标志分配的 pinned memory 可以被多个 CUDA context 使用。

```cpp
CUDA_CHECK(cudaHostAlloc((void**)&h_A, bytes, cudaHostAllocPortable));
```

适合多 GPU、多 CPU 线程或较复杂工程场景。

普通单 GPU 程序通常不需要该标志。

---

## 4.3 `cudaHostAllocMapped`

该标志用于分配可映射的 pinned memory。

```cpp
CUDA_CHECK(cudaHostAlloc((void**)&h_A, bytes, cudaHostAllocMapped));
```

配合：

```cpp
cudaHostGetDevicePointer
```

可以获得 device 端访问该 host memory 的指针，从而实现 zero-copy。

---

## 4.4 `cudaHostAllocWriteCombined`

该标志分配 write-combined memory。

```cpp
CUDA_CHECK(cudaHostAlloc((void**)&h_A, bytes, cudaHostAllocWriteCombined));
```

它适合 CPU 端只写、GPU 端读取的数据传输场景。

特点：

1. host 到 device 传输可能更快。
2. CPU 读取这类内存可能较慢。
3. 适合 host 端生成输入数据后传给 GPU。
4. 不适合 CPU 频繁读取的 host buffer。

---

## 五、pinned memory 与数据传输

## 5.1 同步传输

pinned memory 可以用于普通同步传输：

```cpp
CUDA_CHECK(cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice));
```

即使使用 `cudaMemcpy`，pinned memory 通常也能提升 host-device 传输效率。

---

## 5.2 异步传输

pinned memory 更重要的作用是支持异步传输：

```cpp
CUDA_CHECK(cudaMemcpyAsync(d_A,
                           h_A,
                           bytes,
                           cudaMemcpyHostToDevice,
                           stream));
```

如果 host memory 不是 pinned memory，`cudaMemcpyAsync` 可能无法真正异步，或者行为退化为类似同步传输。

因此，使用异步传输时应遵循：

```text
cudaMemcpyAsync + pinned memory + stream
```

---

## 5.3 pinned memory 简单示例

```cpp
#include <cstdio>
#include <cstdlib>
#include <cuda_runtime.h>

#define CUDA_CHECK(call)                                                   \
    do {                                                                   \
        cudaError_t err = (call);                                           \
        if (err != cudaSuccess) {                                           \
            fprintf(stderr, "CUDA error at %s:%d: %s\n",                   \
                    __FILE__, __LINE__, cudaGetErrorString(err));           \
            exit(EXIT_FAILURE);                                             \
        }                                                                  \
    } while (0)

int main() {
    const int n = 1 << 20;
    const size_t bytes = n * sizeof(float);

    float* h_A = nullptr;
    float* d_A = nullptr;

    CUDA_CHECK(cudaMallocHost((void**)&h_A, bytes));
    CUDA_CHECK(cudaMalloc((void**)&d_A, bytes));

    for (int i = 0; i < n; ++i) {
        h_A[i] = static_cast<float>(i);
    }

    CUDA_CHECK(cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice));

    CUDA_CHECK(cudaFree(d_A));
    CUDA_CHECK(cudaFreeHost(h_A));

    return EXIT_SUCCESS;
}
```

编译命令：

```bash
nvcc -O2 pinned_basic.cu -o pinned_basic
```

## 六、zero-copy

## 6.1 zero-copy 的概念

zero-copy 是指 GPU kernel 直接访问被映射的 host pinned memory，而不需要显式把数据复制到 device memory。

普通流程：

```text
host memory
    ↓ cudaMemcpy
device memory
    ↓ kernel 访问
```

zero-copy 流程：

```text
mapped pinned host memory
    ↓
GPU kernel 直接访问
```

zero-copy 的优点：

1. 不需要显式 `cudaMemcpy`。
2. 可以节省 device memory。
3. 对少量数据或只访问一次的数据可能有利。
4. 简化部分数据传输流程。

zero-copy 的缺点：

1. GPU 访问 host memory 延迟高。
2. 带宽通常低于访问 device global memory。
3. 不适合 GPU 反复访问同一数据。
4. 不适合大规模高频访存计算。
5. 性能依赖硬件和访问模式。

简单判断：

```text
数据只访问一次，且不想显式复制：可以考虑 zero-copy
数据会被 GPU 多次访问：通常复制到 device memory 更好
```

---

## 6.2 检查设备是否支持 mapped memory

可以通过 `cudaGetDeviceProperties` 查询设备属性。

```cpp
cudaDeviceProp prop;
CUDA_CHECK(cudaGetDeviceProperties(&prop, 0));

if (!prop.canMapHostMemory) {
    printf("This device does not support mapped host memory.\n");
}
```

如果设备不支持 mapped host memory，就不能使用 zero-copy 方式让 kernel 直接访问 host memory。

---

## 6.3 启用 host memory mapping

在使用 mapped pinned memory 前，通常需要在创建 CUDA context 前调用：

```cpp
cudaSetDeviceFlags(cudaDeviceMapHost);
```

注意：该调用应尽量放在其他 CUDA Runtime API 调用之前。

示例：

```cpp
CUDA_CHECK(cudaSetDeviceFlags(cudaDeviceMapHost));
CUDA_CHECK(cudaSetDevice(0));
```

如果程序已经隐式创建了 CUDA context，再调用 `cudaSetDeviceFlags` 可能失败或不生效。

---

## 6.4 分配 mapped pinned memory

使用：

```cpp
cudaHostAllocMapped
```

示例：

```cpp
float* h_A = nullptr;
CUDA_CHECK(cudaHostAlloc((void**)&h_A, bytes, cudaHostAllocMapped));
```

然后获取对应的 device pointer：

```cpp
float* d_A = nullptr;
CUDA_CHECK(cudaHostGetDevicePointer((void**)&d_A, h_A, 0));
```

此时：

- `h_A` 是 host pointer。
- `d_A` 是 device pointer。
- 两者指向同一块映射的主机端页锁定内存。
- kernel 中应使用 `d_A`。
- host 端应使用 `h_A`。

---

## 6.5 zero-copy 示例

```cpp
#include <cstdio>
#include <cstdlib>
#include <cuda_runtime.h>

#define CUDA_CHECK(call)                                                   \
    do {                                                                   \
        cudaError_t err = (call);                                           \
        if (err != cudaSuccess) {                                           \
            fprintf(stderr, "CUDA error at %s:%d: %s\n",                   \
                    __FILE__, __LINE__, cudaGetErrorString(err));           \
            exit(EXIT_FAILURE);                                             \
        }                                                                  \
    } while (0)

__global__ void scaleKernel(const float* input, float* output, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    if (idx < n) {
        output[idx] = input[idx] * 2.0f;
    }
}

int main() {
    CUDA_CHECK(cudaSetDeviceFlags(cudaDeviceMapHost));

    int device = 0;
    CUDA_CHECK(cudaSetDevice(device));

    cudaDeviceProp prop;
    CUDA_CHECK(cudaGetDeviceProperties(&prop, device));

    if (!prop.canMapHostMemory) {
        fprintf(stderr, "This device does not support mapped host memory.\n");
        return EXIT_FAILURE;
    }

    const int n = 1 << 20;
    const size_t bytes = n * sizeof(float);

    float* h_input = nullptr;
    float* h_output = nullptr;

    CUDA_CHECK(cudaHostAlloc((void**)&h_input, bytes, cudaHostAllocMapped));
    CUDA_CHECK(cudaHostAlloc((void**)&h_output, bytes, cudaHostAllocMapped));

    for (int i = 0; i < n; ++i) {
        h_input[i] = static_cast<float>(i);
        h_output[i] = 0.0f;
    }

    float* d_input = nullptr;
    float* d_output = nullptr;

    CUDA_CHECK(cudaHostGetDevicePointer((void**)&d_input, h_input, 0));
    CUDA_CHECK(cudaHostGetDevicePointer((void**)&d_output, h_output, 0));

    dim3 block(256);
    dim3 grid((n + block.x - 1) / block.x);

    scaleKernel<<<grid, block>>>(d_input, d_output, n);
    CUDA_CHECK(cudaGetLastError());
    CUDA_CHECK(cudaDeviceSynchronize());

    bool correct = true;
    for (int i = 0; i < n; ++i) {
        float expected = h_input[i] * 2.0f;

        if (h_output[i] != expected) {
            correct = false;
            fprintf(stderr, "Mismatch at %d: got %f, expected %f\n",
                    i, h_output[i], expected);
            break;
        }
    }

    printf("Result: %s\n", correct ? "PASS" : "FAIL");

    CUDA_CHECK(cudaFreeHost(h_input));
    CUDA_CHECK(cudaFreeHost(h_output));

    return correct ? EXIT_SUCCESS : EXIT_FAILURE;
}
```

编译命令：

```bash
nvcc -O2 zero_copy_example.cu -o zero_copy_example
```

运行命令：

```bash
./zero_copy_example
```

## 七、异步执行

## 7.1 同步执行与异步执行

CUDA 中很多操作都是异步的，尤其是 kernel launch。host 端启动 kernel 后，通常不会等待 kernel 执行完成，而是继续执行后续 host code。

同步数据传输函数：

```cpp
cudaMemcpy
```

异步数据传输函数：

```cpp
cudaMemcpyAsync
```

`cudaMemcpy` 的特点：

```text
CPU 线程会等待数据复制完成后才继续向下执行。
```

`cudaMemcpyAsync` 的特点：

```text
在条件满足时，数据复制提交后就可以把控制权返回给 CPU 线程。
```

使用异步传输通常需要：

1. pinned host memory。
2. CUDA stream。
3. 设备支持相关并发能力。
4. 避免全部操作都放在 default stream 中。

---

## 7.2 CUDA stream 的概念

stream 是 CUDA 操作的有序队列。

同一个 stream 中的操作按照提交顺序执行：

```text
H2D copy
    ↓
kernel
    ↓
D2H copy
```

不同 stream 中的操作可能并发执行：

```text
stream 1: H2D copy -> kernel -> D2H copy
stream 2: H2D copy -> kernel -> D2H copy
```

创建 stream：

```cpp
cudaStream_t stream;
CUDA_CHECK(cudaStreamCreate(&stream));
```

销毁 stream：

```cpp
CUDA_CHECK(cudaStreamDestroy(stream));
```

kernel 使用 stream：

```cpp
kernel<<<grid, block, 0, stream>>>(...);
```

异步复制使用 stream：

```cpp
cudaMemcpyAsync(dst, src, bytes, kind, stream);
```

---

## 7.3 default stream

如果不显式指定 stream，CUDA 操作默认使用 0 号流，也称 default stream。

示例：

```cpp
cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, 0);
kernel<<<grid, block>>>(d_A);
```

这两个操作都在 default stream 中，通常会按照顺序执行。也就是说，kernel 会等待前面的 H2D 复制完成。

如果希望数据传输和 kernel 计算重叠，通常需要使用不同的非默认 stream，并且设备支持复制与计算并发。

---

## 八、`cudaMemcpyAsync`

## 8.1 函数形式

```cpp
cudaError_t cudaMemcpyAsync(void* dst,
                            const void* src,
                            size_t count,
                            cudaMemcpyKind kind,
                            cudaStream_t stream);
```

常见方向：

```cpp
cudaMemcpyHostToDevice
cudaMemcpyDeviceToHost
cudaMemcpyDeviceToDevice
```

host 到 device：

```cpp
CUDA_CHECK(cudaMemcpyAsync(d_A,
                           h_A,
                           bytes,
                           cudaMemcpyHostToDevice,
                           stream));
```

device 到 host：

```cpp
CUDA_CHECK(cudaMemcpyAsync(h_C,
                           d_C,
                           bytes,
                           cudaMemcpyDeviceToHost,
                           stream));
```

---

## 8.2 使用条件

为了让 `cudaMemcpyAsync` 真正发挥作用，通常需要满足：

1. host memory 是 pinned memory。
2. 使用非默认 stream。
3. 数据传输和计算之间没有必须串行的依赖。
4. 设备支持拷贝和计算并发。
5. 代码中没有不必要的全局同步。

如果在异步复制后立即调用：

```cpp
cudaDeviceSynchronize();
```

就会破坏异步流水线效果。

---

## 8.3 stream 同步方式

等待某个 stream 完成：

```cpp
CUDA_CHECK(cudaStreamSynchronize(stream));
```

等待整个 device 上所有前序任务完成：

```cpp
CUDA_CHECK(cudaDeviceSynchronize());
```

一般原则：

```text
能同步单个 stream，就不要随意同步整个 device。
```

因为 `cudaDeviceSynchronize()` 会等待设备上所有前序任务完成，可能降低并发效果。

---

## 九、异步传输与 kernel 重叠

## 9.1 host 计算与 GPU 操作重叠

即使使用 default stream，host 端代码也可以与 GPU 异步操作重叠。

示意：

```cpp
cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, 0);
kernel<<<grid, block>>>(d_A);

cpuFunction();

cudaDeviceSynchronize();
```

这里：

1. H2D copy 和 kernel 在 default stream 中顺序执行。
2. CPU 端 `cpuFunction()` 可以在 GPU 执行期间运行。
3. 最后通过 `cudaDeviceSynchronize()` 等待 GPU 完成。

---

## 9.2 数据传输与 kernel 计算重叠

如果设备支持复制与计算并发，可以将数据传输和 kernel 计算放在不同的非默认 stream 中。

示意：

```cpp
cudaStream_t stream1;
cudaStream_t stream2;

cudaStreamCreate(&stream1);
cudaStreamCreate(&stream2);

cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream1);
kernel<<<grid, block, 0, stream2>>>(d_B);

cudaStreamSynchronize(stream1);
cudaStreamSynchronize(stream2);

cudaStreamDestroy(stream1);
cudaStreamDestroy(stream2);
```

这里，`stream1` 中的数据复制和 `stream2` 中的 kernel 可能重叠执行。

注意：如果 kernel 依赖刚复制的数据，就不能简单放到不同 stream 中并发执行，否则会产生数据依赖错误。

---

## 9.3 分块流水线

异步执行最常见的应用是分块流水线处理大数据。

基本思想：

```text
将大数据分成多个 chunk
每个 chunk 依次执行：
    host to device
    kernel
    device to host

多个 chunk 分布在多个 stream 中交错执行
```

这样可以实现：

```text
当前 chunk 正在计算
下一个 chunk 正在传输
上一个 chunk 正在回传
```

从而隐藏部分数据传输开销。

---

## 十、完整示例：stream + pinned memory + cudaMemcpyAsync

下面示例展示如何使用 pinned memory、两个 stream 和 `cudaMemcpyAsync` 处理大数组。

```cpp
#include <cstdio>
#include <cstdlib>
#include <cmath>
#include <cuda_runtime.h>

#define CUDA_CHECK(call)                                                   \
    do {                                                                   \
        cudaError_t err = (call);                                           \
        if (err != cudaSuccess) {                                           \
            fprintf(stderr, "CUDA error at %s:%d: %s\n",                   \
                    __FILE__, __LINE__, cudaGetErrorString(err));           \
            exit(EXIT_FAILURE);                                             \
        }                                                                  \
    } while (0)

__global__ void scaleKernel(const float* input, float* output, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    if (idx < n) {
        output[idx] = input[idx] * 2.0f;
    }
}

int main() {
    const int n = 1 << 24;
    const int chunkSize = 1 << 20;
    const int numStreams = 2;

    const size_t totalBytes = n * sizeof(float);
    const size_t chunkBytes = chunkSize * sizeof(float);

    float* h_input = nullptr;
    float* h_output = nullptr;

    CUDA_CHECK(cudaMallocHost((void**)&h_input, totalBytes));
    CUDA_CHECK(cudaMallocHost((void**)&h_output, totalBytes));

    for (int i = 0; i < n; ++i) {
        h_input[i] = static_cast<float>(i);
        h_output[i] = 0.0f;
    }

    cudaStream_t streams[numStreams];

    float* d_input[numStreams];
    float* d_output[numStreams];

    for (int i = 0; i < numStreams; ++i) {
        CUDA_CHECK(cudaStreamCreate(&streams[i]));
        CUDA_CHECK(cudaMalloc((void**)&d_input[i], chunkBytes));
        CUDA_CHECK(cudaMalloc((void**)&d_output[i], chunkBytes));
    }

    dim3 block(256);

    for (int offset = 0; offset < n; offset += chunkSize) {
        int streamId = (offset / chunkSize) % numStreams;
        int currentChunkSize = chunkSize;

        if (offset + currentChunkSize > n) {
            currentChunkSize = n - offset;
        }

        size_t currentBytes = currentChunkSize * sizeof(float);

        CUDA_CHECK(cudaMemcpyAsync(d_input[streamId],
                                   h_input + offset,
                                   currentBytes,
                                   cudaMemcpyHostToDevice,
                                   streams[streamId]));

        dim3 grid((currentChunkSize + block.x - 1) / block.x);

        scaleKernel<<<grid, block, 0, streams[streamId]>>>(
            d_input[streamId],
            d_output[streamId],
            currentChunkSize
        );

        CUDA_CHECK(cudaGetLastError());

        CUDA_CHECK(cudaMemcpyAsync(h_output + offset,
                                   d_output[streamId],
                                   currentBytes,
                                   cudaMemcpyDeviceToHost,
                                   streams[streamId]));
    }

    for (int i = 0; i < numStreams; ++i) {
        CUDA_CHECK(cudaStreamSynchronize(streams[i]));
    }

    bool correct = true;
    for (int i = 0; i < n; ++i) {
        float expected = h_input[i] * 2.0f;

        if (fabs(h_output[i] - expected) > 1e-5f) {
            correct = false;
            fprintf(stderr, "Mismatch at %d: got %f, expected %f\n",
                    i, h_output[i], expected);
            break;
        }
    }

    printf("Result: %s\n", correct ? "PASS" : "FAIL");

    for (int i = 0; i < numStreams; ++i) {
        CUDA_CHECK(cudaFree(d_input[i]));
        CUDA_CHECK(cudaFree(d_output[i]));
        CUDA_CHECK(cudaStreamDestroy(streams[i]));
    }

    CUDA_CHECK(cudaFreeHost(h_input));
    CUDA_CHECK(cudaFreeHost(h_output));

    return correct ? EXIT_SUCCESS : EXIT_FAILURE;
}
```

编译命令：

```bash
nvcc -O2 async_stream_pinned.cu -o async_stream_pinned
```

运行命令：

```bash
./async_stream_pinned
```

预期输出：

```text
Result: PASS
```

## 十一、查询设备异步能力

不同 GPU 对异步复制和并发执行的支持不同。可以使用 `cudaGetDeviceProperties` 查询设备属性。

示例：

```cpp
#include <cstdio>
#include <cuda_runtime.h>

int main() {
    int device = 0;
    cudaDeviceProp prop;

    cudaGetDeviceProperties(&prop, device);

    printf("Device: %s\n", prop.name);
    printf("canMapHostMemory: %d\n", prop.canMapHostMemory);
    printf("deviceOverlap: %d\n", prop.deviceOverlap);
    printf("asyncEngineCount: %d\n", prop.asyncEngineCount);

}
```

常见属性含义：

| 属性 | 含义 |
|---|---|
| `canMapHostMemory` | 是否支持 mapped host memory |
| `deviceOverlap` | 是否支持复制和 kernel 执行重叠 |
| `asyncEngineCount` | 异步复制引擎数量 |
| `concurrentKernels` | 是否支持并发 kernel |

编译：

```bash
nvcc device_async_info.cu -o device_async_info
```

## 十二、pinned memory 的适用场景

适合使用 pinned memory 的场景：

1. 大规模 host-device 数据传输。
2. 重复进行 host-device 数据传输。
3. 使用 `cudaMemcpyAsync`。
4. 使用 CUDA stream。
5. 希望数据传输和 kernel 计算重叠。
6. 分块流水线处理大数组。
7. 需要 zero-copy 的小规模访问。
8. 数据传输成为主要性能瓶颈。

不适合使用 pinned memory 的场景：

1. 数据量很小。
2. 只运行一次的小程序。
3. host memory 紧张。
4. CPU 频繁读取 `cudaHostAllocWriteCombined` 分配的内存。
5. GPU 会反复访问同一批数据，此时通常复制到 device memory 更好。
6. 程序没有异步传输或传输瓶颈。

## 十三、pinned memory 使用原则

## 13.1 不要过量使用

pinned memory 不会被操作系统换出，因此过量使用会减少系统可调度的物理内存，可能导致系统整体性能下降。

原则：

```text
只给传输热点 buffer 使用 pinned memory。
```

不要把所有 host 数据都改成 pinned memory。

---

## 13.2 优先用于大块连续传输

pinned memory 适合大块数据传输。

不推荐：

```text
大量小块频繁 cudaMemcpyAsync
```

更推荐：

```text
合并为较大的连续 buffer 后传输
```

---

## 13.3 与 stream 配合使用

如果只是使用：

```cpp
cudaMemcpy
```

pinned memory 可能提升传输速度，但不能充分体现异步优势。

更典型的用法是：

```cpp
cudaMemcpyAsync + stream + pinned memory
```

---

## 13.4 注意数据依赖

不同 stream 中的操作可能并发执行，但不能忽略数据依赖。

错误示意：

```cpp
cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream1);
kernel<<<grid, block, 0, stream2>>>(d_A);
```

如果 kernel 依赖 `d_A` 中刚复制的数据，但复制和 kernel 在不同 stream 中，kernel 可能在复制完成前启动。

正确做法：

1. 将有依赖的复制和 kernel 放在同一个 stream 中。
2. 或使用 CUDA event 建立跨 stream 依赖。

简单做法：

```cpp
cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream);
kernel<<<grid, block, 0, stream>>>(d_A);
```

同一个 stream 中操作会按顺序执行。

---

## 13.5 避免不必要的 `cudaDeviceSynchronize`

如果在每个异步操作后都调用：

```cpp
cudaDeviceSynchronize();
```

就会破坏异步执行效果。

低效写法：

```cpp
cudaMemcpyAsync(..., stream);
cudaDeviceSynchronize();

kernel<<<grid, block, 0, stream>>>(...);
cudaDeviceSynchronize();
```

更好写法：

```cpp
cudaMemcpyAsync(..., stream);
kernel<<<grid, block, 0, stream>>>(...);
cudaMemcpyAsync(..., stream);

cudaStreamSynchronize(stream);
```

## 十四、pinned memory 与 zero-copy 的选择

| 场景 | 推荐方式 |
|---|---|
| 数据量大，GPU 多次访问 | 复制到 device global memory |
| 数据量大，需要传输优化 | pinned memory + cudaMemcpyAsync |
| 数据只访问一次，且希望省去复制 | zero-copy |
| 数据需要 GPU 高频访问 | 不推荐 zero-copy |
| 数据传输与计算可流水线化 | pinned memory + stream |
| host 端生成数据，GPU 端读取 | pinned memory 或 write-combined pinned memory |

简单原则：

```text
zero-copy 省去了显式复制，但 GPU 访问 host memory 通常较慢。
pinned memory + cudaMemcpyAsync 更适合高吞吐传输。
device global memory 更适合 GPU 反复访问。
```

## 十五、常见错误

## 15.1 使用 pageable memory 调用 `cudaMemcpyAsync`

错误或低效写法：

```cpp
float* h_A = static_cast<float*>(malloc(bytes));

cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream);
```

如果希望真正异步，应使用 pinned memory：

```cpp
float* h_A = nullptr;
cudaMallocHost((void**)&h_A, bytes);

cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream);
```

---

## 15.2 使用 `free` 释放 pinned memory

错误：

```cpp
cudaMallocHost((void**)&h_A, bytes);
free(h_A);
```

正确：

```cpp
cudaFreeHost(h_A);
```

---

## 15.3 使用 `cudaFreeHost` 释放普通内存

错误：

```cpp
float* h_A = static_cast<float*>(malloc(bytes));
cudaFreeHost(h_A);
```

正确：

```cpp
free(h_A);
```

---

## 15.4 过量分配 pinned memory

错误做法：

```text
将所有 host 数组都分配为 pinned memory。
```

正确做法：

```text
只对大规模传输、异步传输或 zero-copy 所需的 host buffer 使用 pinned memory。
```

---

## 15.5 zero-copy 中使用 host pointer 传给 kernel

错误：

```cpp
scaleKernel<<<grid, block>>>(h_input, h_output, n);
```

即使 `h_input` 是 mapped pinned memory，也不应直接把 host pointer 当成 kernel 参数。

正确做法：

```cpp
cudaHostGetDevicePointer((void**)&d_input, h_input, 0);
cudaHostGetDevicePointer((void**)&d_output, h_output, 0);

scaleKernel<<<grid, block>>>(d_input, d_output, n);
```

---

## 15.6 忘记调用 `cudaSetDeviceFlags(cudaDeviceMapHost)`

如果要使用 mapped memory，通常需要在创建 CUDA context 前调用：

```cpp
cudaSetDeviceFlags(cudaDeviceMapHost);
```

应尽量把它放在其他 CUDA Runtime API 调用之前。

---

## 15.7 忽略设备能力

不是所有设备都支持 mapped host memory 或复制计算并发。

应查询：

```cpp
prop.canMapHostMemory
prop.deviceOverlap
prop.asyncEngineCount
```

---

## 15.8 不恰当地跨 stream 使用数据

错误：

```cpp
cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream1);
kernel<<<grid, block, 0, stream2>>>(d_A);
```

如果 kernel 依赖 `d_A` 中的数据，这样可能存在数据竞争。

正确：

```cpp
cudaMemcpyAsync(d_A, h_A, bytes, cudaMemcpyHostToDevice, stream1);
kernel<<<grid, block, 0, stream1>>>(d_A);
```

或使用 CUDA event 建立依赖。

---

## 15.9 立刻同步导致异步失效

错误：

```cpp
cudaMemcpyAsync(..., stream);
cudaDeviceSynchronize();
```

这样会让异步传输失去意义。

更好：

```cpp
cudaMemcpyAsync(..., stream);
// 提交更多异步任务
cudaStreamSynchronize(stream);
```

## 十六、回答模板

## 16.1 解释 pinned memory

```text
pinned memory 是主机端页锁定内存，它分配在主机端物理内存中，不会被操作系统换出到磁盘。与普通 pageable memory 相比，pinned memory 与 GPU device memory 之间的数据传输通常更高效，并且是 cudaMemcpyAsync 异步传输的重要前提。CUDA 中可以使用 cudaMallocHost 或 cudaHostAlloc 分配 pinned memory，并使用 cudaFreeHost 释放。
```

---

## 16.2 解释 pinned memory 和 pageable memory 的区别

```text
pageable memory 是普通主机内存，通常由 malloc、new 或 std::vector 分配，可能被操作系统换出。pinned memory 是页锁定主机内存，不会被换出，因此 GPU 可以更高效地与其进行数据传输。pageable memory 使用方便，适合普通程序；pinned memory 适合大规模数据传输、cudaMemcpyAsync、stream 和 zero-copy，但不应过量使用。
```

---

## 16.3 解释 zero-copy

```text
zero-copy 是指 GPU kernel 直接访问映射后的主机端页锁定内存，而不需要显式调用 cudaMemcpy 将数据复制到 device global memory。它可以节省显存空间并减少显式数据复制，但 GPU 访问 host memory 的延迟通常高于访问 device memory，因此更适合数据只访问一次或访问量较小的场景，不适合 GPU 反复读取的大规模计算。
```

---

## 16.4 解释 `cudaMemcpyAsync`

```text
cudaMemcpyAsync 是 CUDA 的异步数据传输函数。与 cudaMemcpy 不同，它可以在数据复制提交后将控制权返回给 CPU 线程，使 CPU 可以继续执行其他任务。为了让 cudaMemcpyAsync 真正发挥异步传输效果，host memory 通常需要使用 pinned memory，并且需要配合 CUDA stream 使用。
```

---

## 16.5 解释 stream

```text
stream 是 CUDA 中的操作队列。同一个 stream 中的操作按照提交顺序执行，不同 stream 中的操作可能并发执行。通过将数据传输和 kernel 放入不同 stream，或者将大数据分块放入多个 stream，可以实现数据传输与 GPU 计算的重叠，从而隐藏部分传输延迟。
```

---

## 16.6 解释为什么 pinned memory 不能过量使用

```text
pinned memory 不会被操作系统换出，会长期占用物理内存。如果大量分配 pinned memory，操作系统和其他程序可用的物理内存会减少，可能导致系统整体性能下降。因此 pinned memory 应只用于大规模传输、异步传输或 zero-copy 等性能关键的 host buffer，而不应替代所有普通 host memory。
```

## 十八、小结

主机端页锁定内存 pinned memory 是 CUDA 中优化 host-device 数据传输的重要工具。与普通 pageable memory 不同，pinned memory 分配在物理内存中，不会被操作系统换出，因此 GPU 与其之间的数据传输通常更高效。CUDA 中可以使用 `cudaMallocHost` 或 `cudaHostAlloc` 分配 pinned memory，并使用 `cudaFreeHost` 释放。

pinned memory 的两个重要应用是 zero-copy 和异步执行。zero-copy 通过 mapped pinned memory 让 GPU kernel 直接访问 host memory，可以节省显式数据复制和 device memory 空间，但由于 GPU 访问 host memory 延迟较高，它更适合访问次数少的数据。异步执行则通过 `cudaMemcpyAsync` 和 stream 实现数据传输与 kernel 执行的重叠，适合大规模数据分块流水线处理。

本节的核心结论是：pinned memory 不是普通 host memory 的无条件替代品，而是用于传输优化、异步复制和 zero-copy 的性能工具。使用时应关注数据规模、访问次数、设备能力、stream 依赖关系和系统物理内存压力。
