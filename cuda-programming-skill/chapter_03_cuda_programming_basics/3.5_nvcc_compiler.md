# 3.5 nvcc 编译器

## 一、本节作用

本节用于整理 CUDA 编译器 `nvcc` 的核心知识。前面的 CUDA 编程模型解决的是“CUDA 程序如何组织”和“kernel 如何启动”的问题，而本节解决的是“CUDA 程序如何被编译和运行”的问题。

CUDA 程序通常保存在 `.cu` 文件中，其中既包含运行在 CPU 上的 host code，也包含运行在 GPU 上的 device code。普通 C/C++ 编译器不能直接处理 CUDA 的扩展语法，例如：

```cpp
__global__
__device__
__shared__
kernel<<<grid, block>>>(...)
```

因此，CUDA 程序需要使用 NVIDIA 提供的 `nvcc` 编译器进行编译。

掌握本节后，应能够完成以下任务：

1. 使用 `nvcc` 编译 `.cu` 文件。
2. 理解 CUDA 程序中 host code 和 device code 的编译过程。
3. 理解 PTX、cubin 和可执行文件之间的关系。
4. 使用 `-arch`、`-code`、`-gencode` 等参数指定 GPU 架构。
5. 使用 `-O2`、`-G`、`-lineinfo` 等选项控制优化与调试。
6. 处理常见 CUDA 编译错误。
7. 根据 GPU 计算能力选择合适的编译参数。

---

## 二、nvcc 的基本作用

`nvcc` 是 NVIDIA CUDA Compiler 的缩写，是 CUDA Toolkit 提供的编译驱动程序。

它不是单纯的 C/C++ 编译器，而是一个编译流程管理工具。它会根据 CUDA 源文件中的代码类型，将 host code 和 device code 分别交给不同的编译组件处理。

简单来说，`nvcc` 的作用包括：

1. 识别 `.cu` 文件。
2. 分离 host code 和 device code。
3. 将 host code 交给主机端 C/C++ 编译器处理。
4. 将 device code 编译为 PTX 或 cubin。
5. 将 GPU 端代码嵌入或链接到最终程序中。
6. 生成可执行文件、目标文件或中间文件。

最常用的编译命令如下：

```bash
nvcc main.cu -o main
```

使用优化选项：

```bash
nvcc -O2 main.cu -o main
```

使用 C++17 标准：

```bash
nvcc -O2 -std=c++17 main.cu -o main
```

---

## 三、CUDA 程序的编译对象

一个 CUDA 源文件中通常包含两类代码。

### 1. host code

host code 是运行在 CPU 上的代码。

例如：

```cpp
int main() {
    cudaMalloc(...);
    cudaMemcpy(...);
    kernel<<<grid, block>>>(...);
    cudaFree(...);
    return 0;
}
```

这部分代码最终由主机端 C/C++ 编译器编译，例如 Linux 下的 `gcc` / `g++`，Windows 下的 MSVC。

---

### 2. device code

device code 是运行在 GPU 上的代码。

例如：

```cpp
__global__ void kernel(float* data) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    data[idx] = data[idx] * 2.0f;
}
```

这部分代码由 CUDA 设备端编译流程处理，最终生成 GPU 可执行的代码。

---

## 四、nvcc 编译流程

教材中将 `nvcc` 的基本编译流程概括为四个主要步骤：

1. 分离主机代码和设备代码。
2. 将主机代码输出为 C/C++ 代码，或调用主机编译器生成目标代码。
3. 将设备代码编译为 PTX 汇编形式，或编译为 cubin 二进制形式。
4. 通过 CUDA 驱动 API 或链接方式，将 PTX / cubin 装载到主机程序中执行。

可以抽象为：

```text
.cu source file
    |
    |-- host code  ------> host compiler ------> object code
    |
    |-- device code -----> PTX / cubin --------> embedded into host program
    |
    v
executable file
```

更细一点可以理解为：

```text
main.cu
  |
  | 1. 分离 host code 与 device code
  |
  | 2. host code 交给 C/C++ 编译器
  |
  | 3. device code 生成 PTX
  |
  | 4. PTX 进一步生成 cubin 或保留为中间表示
  |
  | 5. GPU 代码被打包进最终程序
  |
  v
main
```

---

## 五、PTX 与 cubin

理解 `nvcc` 编译流程时，需要区分 PTX 和 cubin。

---

### 1. PTX

PTX 是 Parallel Thread Execution 的缩写，可以理解为 CUDA 设备代码的一种中间汇编表示。

PTX 的特点：

1. 不是最终机器码。
2. 具有一定的跨 GPU 架构兼容性。
3. 可以在运行时由驱动进一步编译为具体 GPU 可执行的机器码。
4. 适合用于提高向后兼容性。

生成 PTX 的命令示例：

```bash
nvcc -ptx main.cu -o main.ptx
```

---

### 2. cubin

cubin 是 CUDA binary 的缩写，是面向具体 GPU 架构生成的二进制代码。

cubin 的特点：

1. 更接近最终 GPU 执行代码。
2. 与具体的 `sm_xx` 架构相关。
3. 对目标 GPU 架构要求更明确。
4. 通常性能确定性更强，但兼容范围更窄。

生成 cubin 的命令示例：

```bash
nvcc -cubin main.cu -o main.cubin
```

---

### 3. 二者关系

可以简单理解为：

```text
CUDA C/C++ source
        ↓
      PTX
        ↓
      cubin
        ↓
   GPU execution
```

PTX 偏向中间表示，cubin 偏向特定 GPU 架构的二进制表示。

---

## 六、使用 `-keep` 查看中间文件

`nvcc` 默认会隐藏大部分中间编译文件。如果希望观察 CUDA 编译过程，可以使用 `-keep` 选项保留中间文件。

示例：

```bash
nvcc -keep main.cu -o main
```

教材示例中使用类似命令：

```bash
nvcc -cuda version.cu -keep
```

编译后会生成一系列中间文件，例如：

```text
version.ptx
version.sm_10.cubin
version.cu.cpp
version.fatbin.c
version.cudafe1.cpp
version.cudafe1.stub.c
```

其中比较重要的文件包括：

| 文件 | 作用 |
|---|---|
| `.cu` | 原始 CUDA 源文件 |
| `.ptx` | 设备端 PTX 中间代码 |
| `.cubin` | 面向特定 GPU 架构的二进制设备代码 |
| `.cu.cpp` | 分离并处理后的 host 端 C++ 代码 |
| `.fatbin.c` | 打包后的设备端代码数据 |

使用 `-keep` 的意义是帮助学习者理解 `nvcc` 并不是简单地“一步编译”，而是经过了 host/device 分离、设备代码编译、打包和链接等多个阶段。

## 七、最常用 nvcc 编译命令

### 1. 编译单个 `.cu` 文件

```bash
nvcc main.cu -o main
```

---

### 2. 使用优化选项

```bash
nvcc -O2 main.cu -o main
```

或：

```bash
nvcc -O3 main.cu -o main
```

一般课程实验中常用 `-O2`，因为它已经能启用较多优化，同时编译时间和调试复杂度相对适中。

---

### 3. 指定 C++ 标准

```bash
nvcc -std=c++17 main.cu -o main
```

结合优化：

```bash
nvcc -O2 -std=c++17 main.cu -o main
```

---

### 4. 只编译不链接

```bash
nvcc -c main.cu -o main.o
```

适合多文件工程。

---

### 5. 生成 PTX

```bash
nvcc -ptx main.cu -o main.ptx
```

---

### 6. 生成 cubin

```bash
nvcc -cubin main.cu -o main.cubin
```

---

### 7. 保留中间文件

```bash
nvcc -keep main.cu -o main
```

---

### 8. 查看详细编译过程

```bash
nvcc -v main.cu -o main
```

`-v` 可以输出更详细的编译命令和调用过程，适合排查编译链问题。

---

### 9. 生成调试版本

```bash
nvcc -g -G main.cu -o main_debug
```

其中：

- `-g` 用于 host 端调试信息。
- `-G` 用于 device 端调试信息。

注意：`-G` 会关闭或影响很多 device 端优化，因此不能用 `-G` 版本代表真实性能。

---

### 10. 保留行号信息

```bash
nvcc -O2 -lineinfo main.cu -o main
```

`-lineinfo` 常用于性能分析工具，使 profiling 结果能够对应到源代码行。

## 八、GPU 架构参数

CUDA 编译时经常需要指定 GPU 架构。常见参数包括：

```bash
-arch
-code
-gencode
```

---

### 1. `-arch`

`-arch` 用于指定虚拟计算架构，通常写作：

```bash
-arch=compute_XX
```

也可以在简单场景中写作：

```bash
-arch=sm_XX
```

例如：

```bash
nvcc -O2 -arch=sm_75 main.cu -o main
```

这里的 `sm_75` 表示面向计算能力 7.5 的 GPU 架构生成代码。

---

### 2. `-code`

`-code` 用于指定真实 GPU 架构代码，例如：

```bash
-code=sm_75
```

它通常与 `-arch=compute_75` 搭配使用：

```bash
nvcc -O2 -arch=compute_75 -code=sm_75 main.cu -o main
```

---

### 3. `-gencode`

`-gencode` 是更灵活的架构指定方式，适合同时生成多个架构版本。

示例：

```bash
nvcc -O2 main.cu -o main \
    -gencode arch=compute_75,code=sm_75 \
    -gencode arch=compute_86,code=sm_86
```

也可以同时包含 PTX，以提高未来架构兼容性：

```bash
nvcc -O2 main.cu -o main \
    -gencode arch=compute_86,code=sm_86 \
    -gencode arch=compute_86,code=compute_86
```

其中：

```text
code=sm_86
```

表示生成针对具体 GPU 架构的二进制代码。

```text
code=compute_86
```

表示保留 PTX，用于未来可能的 JIT 编译。

---

## 九、如何查看 GPU 计算能力

在编译 CUDA 程序前，最好知道当前 GPU 的计算能力。

---

### 1. 使用 `nvidia-smi`

```bash
nvidia-smi
```

`nvidia-smi` 可以查看 GPU 型号、驱动版本、显存占用等信息，但不一定直接显示计算能力。

---

### 2. 使用 CUDA 示例 `deviceQuery`

CUDA Toolkit 通常提供 `deviceQuery` 示例程序。运行后可以看到类似信息：

```text
CUDA Capability Major/Minor version number: 8.6
```

这表示当前 GPU 的计算能力为 8.6，对应编译参数可以写作：

```bash
-arch=sm_86
```

---

### 3. 在程序中查询

可以使用 `cudaGetDeviceProperties` 查询：

```cpp
#include <cstdio>
#include <cuda_runtime.h>

int main() {
    int count = 0;
    cudaGetDeviceCount(&count);

    for (int i = 0; i < count; ++i) {
        cudaDeviceProp prop;
        cudaGetDeviceProperties(&prop, i);

        printf("Device %d: %s\n", i, prop.name);
        printf("Compute capability: %d.%d\n", prop.major, prop.minor);
    }

    return 0;
}
```

编译：

```bash
nvcc device_info.cu -o device_info
```

运行：

```bash
./device_info
```

## 十、兼容性分析

CUDA 编译中的兼容性主要包括：

1. 二进制兼容性。
2. PTX 兼容性。
3. CUDA Toolkit 与 GPU 驱动兼容性。
4. CUDA Toolkit 与主机编译器兼容性。
5. 程序使用的 CUDA 特性与 GPU 计算能力兼容性。

---

### 1. 二进制兼容性

二进制 cubin 通常面向具体 GPU 架构生成，例如：

```text
sm_75
sm_86
sm_89
```

如果程序只包含某个具体架构的 cubin，那么它通常只能在兼容该架构的 GPU 上运行。

教材中以较早的计算能力为例，列出过 `sm10`、`sm11`、`sm12`、`sm13` 等参数，并指出不同参数对应不同硬件能力。例如，`sm13` 相比之前架构增加了对双精度浮点运算的支持。因此，如果程序使用双精度等特性，就需要目标 GPU 具备对应的计算能力。

现代 CUDA 中，架构参数通常写作：

```text
sm_50
sm_60
sm_70
sm_75
sm_80
sm_86
sm_89
sm_90
```

具体可用值取决于 CUDA Toolkit 版本。

---

### 2. PTX 兼容性

PTX 是一种中间表示。相比 cubin，PTX 通常具有更好的未来兼容性。驱动可以在运行时将 PTX JIT 编译为当前 GPU 可执行的机器码。

例如：

```bash
nvcc -O2 main.cu -o main \
    -gencode arch=compute_86,code=sm_86 \
    -gencode arch=compute_86,code=compute_86
```

这里既包含 `sm_86` 二进制代码，也保留 `compute_86` PTX。这样程序在目标架构上可以直接运行，在某些未来架构上也可能通过 PTX JIT 编译运行。

---

### 3. CUDA Toolkit 与驱动兼容性

CUDA 程序运行依赖 NVIDIA 驱动。通常 CUDA Toolkit 版本越新，对驱动版本要求也越高。

如果驱动版本过低，可能出现类似错误：

```text
CUDA driver version is insufficient for CUDA runtime version
```

解决方法通常是：

1. 更新 NVIDIA 驱动。
2. 使用与当前驱动兼容的 CUDA Toolkit。
3. 在服务器环境中加载正确的 CUDA module。

---

### 4. CUDA Toolkit 与主机编译器兼容性

`nvcc` 编译 host code 时会调用主机端 C/C++ 编译器。因此，CUDA Toolkit 对 `gcc`、`g++` 或 MSVC 的版本有兼容性要求。

常见错误包括：

```text
unsupported GNU version
```

或：

```text
Host compiler targets unsupported OS
```

解决方法包括：

1. 更换 CUDA 支持的主机编译器版本。
2. 使用系统中指定版本的 `gcc` / `g++`。
3. 在命令中通过 `-ccbin` 指定主机编译器。
4. 更新或降低 CUDA Toolkit 版本。

示例：

```bash
nvcc -ccbin g++-10 main.cu -o main
```

---

### 5. 程序特性与计算能力兼容性

某些 CUDA 特性需要特定计算能力支持。例如：

1. 某些原子操作需要较高计算能力。
2. 双精度性能和支持情况与 GPU 架构有关。
3. 动态并行、统一内存、某些协作组功能都有计算能力要求。
4. 较新的 CUDA API 可能要求较新的 Toolkit 和驱动。

如果程序使用了目标 GPU 不支持的特性，可能会出现编译错误或运行错误。

## 十一、编译与运行示例

### 1. HelloWorld 示例

源文件：

```text
hello_world.cu
```

代码：

```cpp
#include <cstdio>
#include <cuda_runtime.h>

__global__ void helloKernel() {
    printf("Hello from GPU!\n");
}

int main() {
    printf("Hello from CPU!\n");

    helloKernel<<<1, 1>>>();

    cudaDeviceSynchronize();

    return 0;
}
```

编译：

```bash
nvcc hello_world.cu -o hello_world
```

运行：

```bash
./hello_world
```

输出：

```text
Hello from CPU!
Hello from GPU!
```

指定架构：

## 十二、多文件 CUDA 工程编译

当程序较大时，可能会拆分为多个文件。

例如：

```text
main.cpp
kernel.cu
kernel.h
```

其中：

`kernel.h`：

```cpp
#pragma once

void launchVectorAdd(const float* d_A, const float* d_B, float* d_C, int n);
```

`kernel.cu`：

```cpp
#include "kernel.h"
#include <cuda_runtime.h>

__global__ void vectorAddKernel(const float* A, const float* B, float* C, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    if (idx < n) {
        C[idx] = A[idx] + B[idx];
    }
}

void launchVectorAdd(const float* d_A, const float* d_B, float* d_C, int n) {
    dim3 block(256);
    dim3 grid((n + block.x - 1) / block.x);

    vectorAddKernel<<<grid, block>>>(d_A, d_B, d_C, n);
}
```

编译方式：

```bash
nvcc -O2 main.cpp kernel.cu -o app
```

也可以分步编译：

```bash
nvcc -O2 -c kernel.cu -o kernel.o
g++ -O2 -c main.cpp -o main.o
nvcc main.o kernel.o -o app
```

最后链接时建议仍使用 `nvcc`，因为 CUDA Runtime 库的链接更方便。

## 十三、与 Makefile 配合使用

简单 Makefile 示例：

```makefile
TARGET = main
SRC = main.cu

NVCC = nvcc
NVCC_FLAGS = -O2 -std=c++17

all:
	$(NVCC) $(NVCC_FLAGS) $(SRC) -o $(TARGET)

clean:
	rm -f $(TARGET) *.o *.ptx *.cubin
```

使用方法：

```bash
make
```

清理：

```bash
make clean
```

如果需要指定架构：

```makefile
NVCC_FLAGS = -O2 -std=c++17 -arch=sm_86
```

## 十四、与 CMake 配合使用

现代 CUDA 工程也可以使用 CMake 管理。

`CMakeLists.txt` 示例：

```cmake
cmake_minimum_required(VERSION 3.18)

project(cuda_example LANGUAGES CXX CUDA)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CUDA_STANDARD 17)

add_executable(main main.cu)

set_target_properties(main PROPERTIES
    CUDA_SEPARABLE_COMPILATION ON
)
```

编译：

```bash
mkdir build
cd build
cmake ..
cmake --build .
```

如果需要指定 CUDA 架构，可以加入：

```cmake
set(CMAKE_CUDA_ARCHITECTURES 86)
```

或者：

```cmake
set_property(TARGET main PROPERTY CUDA_ARCHITECTURES 86)
```

## 十五、调试编译选项

### 1. host 和 device 调试

```bash
nvcc -g -G main.cu -o main_debug
```

说明：

- `-g` 生成 host 端调试信息。
- `-G` 生成 device 端调试信息。

注意：`-G` 会显著影响性能，不适合用于性能测试。

---

### 2. 保留行号信息

```bash
nvcc -O2 -lineinfo main.cu -o main
```

`-lineinfo` 对性能分析很有用，能让 profiling 工具将性能热点定位到源代码行。

---

### 3. 显示 register 使用情况

```bash
nvcc -O2 -Xptxas=-v main.cu -o main
```

可能输出类似：

```text
ptxas info    : Used 32 registers, 0 bytes smem, 344 bytes cmem[0]
```

这对分析寄存器占用、shared memory 使用和 occupancy 有帮助。

---

### 4. 关闭优化进行问题定位

```bash
nvcc -O0 -g -G main.cu -o main_debug
```

用于调试逻辑错误。

## 十六、优化编译选项

### 1. 常规优化

```bash
nvcc -O2 main.cu -o main
```

或：

```bash
nvcc -O3 main.cu -o main
```

---

### 2. 快速数学函数

```bash
nvcc -O2 --use_fast_math main.cu -o main
```

`--use_fast_math` 可能使用更快但精度略低的数学函数。适合对精度要求不极端严格的场景。

不应在没有说明精度影响的情况下默认使用该选项。

---

### 3. 限制寄存器数量

```bash
nvcc -O2 -maxrregcount=64 main.cu -o main
```

该选项可以限制每个线程使用的寄存器数量，有时可以提高 occupancy，但也可能造成寄存器溢出到 local memory，反而降低性能。

使用时应结合 profiling 数据分析。

---

## 十七、常见编译错误

### 1. 使用普通 C++ 编译器编译 `.cu` 文件

错误命令：

```bash
g++ main.cu -o main
```

可能错误：

```text
error: expected primary-expression before '<' token
```

原因是 `g++` 不能识别 CUDA kernel 启动语法：

```cpp
kernel<<<grid, block>>>(...);
```

正确命令：

```bash
nvcc main.cu -o main
```

---

### 2. 文件扩展名错误

错误文件名：

```text
main.cpp
```

但文件中包含：

```cpp
__global__ void kernel() {}
kernel<<<1, 1>>>();
```

普通 `.cpp` 文件不会自动按 CUDA 源文件处理。

正确做法：

```text
main.cu
```

并使用：

```bash
nvcc main.cu -o main
```

---

### 3. 忘记包含 CUDA 头文件

如果使用 CUDA Runtime API，建议包含：

```cpp
#include <cuda_runtime.h>
```

如果使用 `printf`，还需要：

```cpp
#include <cstdio>
```

---

### 4. kernel 函数没有使用 `__global__`

错误写法：

```cpp
void kernel() {
}
```

却使用：

```cpp
kernel<<<1, 1>>>();
```

正确写法：

```cpp
__global__ void kernel() {
}
```

---

### 5. kernel 返回值不是 `void`

错误写法：

```cpp
__global__ int kernel() {
    return 0;
}
```

正确写法：

```cpp
__global__ void kernel() {
}
```

CUDA kernel 的返回类型必须是 `void`。

---

### 6. 主机编译器版本不兼容

可能错误：

```text
unsupported GNU version
```

原因是当前 CUDA Toolkit 不支持系统默认 `gcc` / `g++` 版本。

解决方法：

```bash
nvcc -ccbin g++-10 main.cu -o main
```

或者安装与 CUDA Toolkit 兼容的编译器版本。

---

### 7. 架构参数不被当前 nvcc 支持

错误示例：

```bash
nvcc -arch=sm_90 main.cu -o main
```

如果当前 CUDA Toolkit 较旧，可能不认识 `sm_90`。

解决方法：

1. 使用当前 Toolkit 支持的架构参数。
2. 升级 CUDA Toolkit。
3. 查询当前 `nvcc` 支持的架构。

可尝试：

```bash
nvcc --help
```

---

### 8. 架构参数与 GPU 不匹配

如果编译出的程序只包含较高架构的 cubin，而运行 GPU 架构较低，可能无法运行。

解决方法是选择与目标 GPU 匹配的 `sm_xx`，或者同时保留 PTX：

```bash
nvcc -O2 main.cu -o main \
    -gencode arch=compute_75,code=sm_75 \
    -gencode arch=compute_75,code=compute_75
```

---

### 9. 多文件工程链接错误

可能错误：

```text
undefined reference to ...
```

原因可能是：

1. 函数声明与定义不一致。
2. `.cu` 文件没有参与编译。
3. 最后链接时没有使用 `nvcc`。
4. CUDA Runtime 库没有正确链接。

建议最后链接使用：

```bash
nvcc main.o kernel.o -o app
```

---

### 10. 重复定义错误

如果把 `.cu` 文件直接 `#include` 到另一个 `.cu` 或 `.cpp` 中，同时又把该 `.cu` 文件加入工程编译，可能产生重复定义。

错误习惯：

```cpp
#include "kernel.cu"
```

更推荐：

```cpp
#include "kernel.h"
```

并单独编译：

```bash
nvcc -c kernel.cu -o kernel.o
```

---

## 十八、常见运行错误与编译关系

有些错误虽然发生在运行时，但与编译参数有关。

---

### 1. 驱动版本不足

错误：

```text
CUDA driver version is insufficient for CUDA runtime version
```

原因：

```text
当前 NVIDIA 驱动版本太低，不能支持编译时使用的 CUDA Runtime 版本。
```

解决：

1. 更新 NVIDIA 驱动。
2. 使用较低版本 CUDA Toolkit 重新编译。
3. 在服务器环境中加载正确 CUDA module。

---

### 2. 没有可用 CUDA 设备

错误：

```text
no CUDA-capable device is detected
```

可能原因：

1. 机器没有 NVIDIA GPU。
2. 驱动没有正确安装。
3. 当前环境看不到 GPU。
4. 容器没有正确挂载 GPU。

检查命令：

```bash
nvidia-smi
```

---

### 3. 无效设备函数

错误：

```text
invalid device function
```

常见原因是编译目标架构与实际 GPU 不匹配。

例如，为较新架构编译的 cubin 在较旧 GPU 上运行，可能出现该错误。

解决方法是重新指定正确架构：

```bash
nvcc -O2 -arch=sm_75 main.cu -o main
```

具体 `sm_xx` 应根据目标 GPU 的计算能力选择。

## 十九、编译参数选择建议

### 1. 入门学习

```bash
nvcc main.cu -o main
```

适合最小 HelloWorld 和课堂入门示例。

---

### 2. 普通实验

```bash
nvcc -O2 main.cu -o main
```

适合大多数课程实验。

---

### 3. 使用现代 C++ 语法

```bash
nvcc -O2 -std=c++17 main.cu -o main
```

适合包含 `std::vector`、`std::chrono`、模板等 C++ 代码的程序。

---

### 4. 调试程序

```bash
nvcc -g -G main.cu -o main_debug
```

适合排查 kernel 逻辑错误，但不适合性能测试。

### 5. 性能分析

```bash
nvcc -O2 -lineinfo main.cu -o main
```

适合配合 Nsight Systems、Nsight Compute 等工具进行性能分析。

---

### 6. 指定目标 GPU 架构

```bash
nvcc -O2 -arch=sm_86 main.cu -o main
```

适合已知目标 GPU 计算能力的情况。

## 二十、回答模板

当用户询问 `nvcc` 是什么时，可以回答：

```text
nvcc 是 NVIDIA CUDA Toolkit 提供的 CUDA 编译器驱动程序。它负责处理 .cu 文件，将其中的 host code 和 device code 分离。host code 会交给主机端 C/C++ 编译器处理，device code 会被编译为 PTX 或 cubin，最后再打包或链接到可执行程序中。因此，包含 __global__ 函数和 <<<grid, block>>> 启动语法的 CUDA 程序通常需要使用 nvcc 编译。
```

当用户询问如何编译 CUDA 程序时，可以回答：

```text
最基本的编译命令是 nvcc main.cu -o main。如果程序使用 C++17，可以写成 nvcc -O2 -std=c++17 main.cu -o main。如果需要指定 GPU 架构，可以加上 -arch=sm_xx，例如 -arch=sm_86。具体 sm_xx 应根据 GPU 的计算能力选择。
```

当用户询问 PTX 和 cubin 的区别时，可以回答：

```text
PTX 是 CUDA 设备代码的中间表示，具有一定跨架构兼容性，可以由驱动在运行时 JIT 编译为目标 GPU 机器码。cubin 是面向具体 GPU 架构生成的二进制代码，通常与 sm_xx 架构绑定更紧密，兼容范围较窄，但更接近最终执行代码。
```

当用户遇到 `invalid device function` 时，可以回答：

```text
invalid device function 通常说明程序中的 device code 与当前 GPU 架构不匹配。常见原因是编译时指定的 -arch 或 -code 参数高于当前 GPU 的计算能力，或者可执行文件中没有包含适合当前 GPU 的 cubin / PTX。可以使用 deviceQuery 查看 GPU 计算能力，然后重新使用对应的 -arch=sm_xx 编译。
```

当用户遇到 `unsupported GNU version` 时，可以回答：

```text
unsupported GNU version 表示当前 CUDA Toolkit 不支持系统默认的 gcc/g++ 版本。可以安装 CUDA 支持的编译器版本，并通过 nvcc 的 -ccbin 参数指定，例如 nvcc -ccbin g++-10 main.cu -o main。
```

## 二十二、小结

`nvcc` 是 CUDA 程序从源代码变成可执行程序的核心工具。它能够识别 `.cu` 文件中的 CUDA 扩展语法，将 host code 与 device code 分离，并分别交给主机编译器和设备编译流程处理。设备端代码可以被编译为 PTX 中间表示，也可以进一步生成面向具体 GPU 架构的 cubin 二进制代码。

在实际开发中，最常用的命令是：

```bash
nvcc -O2 main.cu -o main
```

如果需要使用现代 C++，可以添加：

```bash
-std=c++17
```

如果需要指定 GPU 架构，可以使用：

```bash
-arch=sm_xx
```

如果需要调试 kernel，可以使用：

```bash
-g -G
```

如果需要性能分析，可以使用：

```bash
-lineinfo
```

学习本节后，应能够独立完成 CUDA 程序的编译、运行和基础编译错误排查。至此，第 3 章中“CUDA 编程模型、nvcc 编译器、HelloWorld 编程实例”三部分已经形成闭环，为后续学习 GPU 存储器使用技巧和 CUDA 程序优化奠定基础。
